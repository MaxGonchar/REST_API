# Development Tools - Testing and Debugging REST APIs

## The Problem: How Do You Test APIs During Development?

When building a REST API, you need to:
- **Send HTTP requests** to test endpoints
- **Inspect responses** (status codes, headers, body)
- **Debug issues** (why is this returning 500?)
- **Share examples** with team members
- **Automate repetitive tests**
- **Work without a UI** (API isn't connected to frontend yet)

**You can't use a web browser for most of this** because browsers:
- Only send GET requests easily
- Can't customize headers easily
- Can't send POST/PUT/DELETE without forms
- Don't show full request/response details

**Solution:** Specialized API development tools that let you craft any HTTP request and inspect every detail of the response.

---

## Tool Categories

### Visual/GUI Tools
- **Postman** - Full-featured GUI application
- **Insomnia** - Lightweight, clean GUI alternative

### Command-Line Tools
- **curl** - Universal, comes pre-installed
- **HTTPie** - Modern, user-friendly curl alternative

## Postman - The Industry Standard

### What Problem It Solves

**Postman is a complete API development platform** for teams building APIs.

**Use cases:**
- Manual API testing during development
- Creating and sharing request collections with team
- Automating test suites
- Generating API documentation
- Monitoring production APIs
- Collaborative API development

### Key Features

#### 1. Visual Request Builder
```
No code needed - build requests with GUI:

┌─────────────────────────────────────┐
│ GET  https://api.example.com/users  │
├─────────────────────────────────────┤
│ Headers:                            │
│  Authorization: Bearer abc123       │
│  Content-Type: application/json     │
├─────────────────────────────────────┤
│ Body:                               │
│  {                                  │
│    "name": "John Doe"               │
│  }                                  │
└─────────────────────────────────────┘
         [Send Button]
```

#### 2. Collections (Organize Requests)
```
Project: E-commerce API
├── Authentication
│   ├── Login
│   ├── Logout
│   └── Refresh Token
├── Users
│   ├── Get User
│   ├── Create User
│   ├── Update User
│   └── Delete User
└── Products
    ├── List Products
    ├── Get Product
    └── Create Product
```

**Explanation:** Collections organize related requests. Share the entire collection with team members.

#### 3. Environment Variables
```javascript
// Define variables per environment
Development:
  BASE_URL: http://localhost:3000
  API_KEY: dev_key_123

Staging:
  BASE_URL: https://staging.example.com
  API_KEY: staging_key_456

Production:
  BASE_URL: https://api.example.com
  API_KEY: prod_key_789

// Use in requests
GET {{BASE_URL}}/users
Authorization: Bearer {{API_KEY}}
```

**Explanation:** Switch between environments with one click. No need to manually change URLs.

#### 4. Pre-request Scripts
```javascript
// Run JavaScript before request
pm.environment.set("timestamp", Date.now());

// Generate dynamic data
pm.environment.set("randomEmail", `user${Math.random()}@example.com`);

// Calculate auth signature
const signature = CryptoJS.HmacSHA256(request.body, pm.environment.get("SECRET"));
pm.environment.set("signature", signature);
```

**Use case:** Automatically generate timestamps, signatures, or random data for each request.

#### 5. Test Scripts (Automated Testing)
```javascript
// Tests run after receiving response
pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});

pm.test("Response time under 200ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(200);
});

pm.test("User has email", function () {
    const jsonData = pm.response.json();
    pm.expect(jsonData.email).to.exist;
});

// Save data for next request
pm.environment.set("userId", pm.response.json().id);
```

**Use case:** Chain requests together, save tokens, validate responses automatically.

#### 6. Mock Servers
```javascript
// Create mock API responses without writing code
GET /users/123 → Returns:
{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com"
}

// Frontend developers can work while backend is being built
```

#### 7. API Documentation Generation
```
Automatically generate docs from collections:
- All endpoints documented
- Example requests/responses
- Can publish to web
- Always up-to-date
```

### Pros and Cons

#### ✅ Advantages

1. **Easy to learn** - Intuitive GUI, no coding required
2. **Comprehensive** - Everything in one tool (testing, docs, monitoring)
3. **Team collaboration** - Share collections, environments
4. **Great for beginners** - Visual, doesn't require command line knowledge
5. **Automated testing** - Can replace dedicated test frameworks
6. **Cross-platform** - Windows, Mac, Linux
7. **Large community** - Tons of tutorials, examples
8. **Mock servers** - Test before backend is ready
9. **Code generation** - Generates code in multiple languages

#### ❌ Disadvantages

1. **Resource heavy** - Electron app, uses significant RAM (500MB+)
2. **Requires account** - Must sign in to use many features
3. **Sync can be slow** - With large collections
4. **Overkill for simple tasks** - Too heavy for quick one-off requests
5. **Not scriptable** - Can't easily integrate into custom workflows
6. **Vendor lock-in** - Collections are Postman-specific format
7. **Free tier limitations** - Team features require paid plan

### When to Choose Postman

✅ **Best for:**
- Teams collaborating on API development
- Creating comprehensive test suites
- Generating API documentation
- Complex workflows with multiple chained requests
- Learning APIs (beginners)
- Manual testing during development
- When you need mock servers

❌ **Avoid when:**
- Making quick one-off requests (use curl)
- Automating in CI/CD (use dedicated test frameworks)
- Resource-constrained environments
- Working offline without account
- Need simple, lightweight tool

### Quick Start Example

```
1. Download Postman from postman.com

2. Create a new request:
   - Method: GET
   - URL: https://jsonplaceholder.typicode.com/users/1
   - Click "Send"

3. View response:
   Status: 200 OK
   Body:
   {
     "id": 1,
     "name": "Leanne Graham",
     "email": "Sincere@april.biz",
     ...
   }

4. Save to collection:
   - Name: "Get User"
   - Collection: "JSON Placeholder Tests"

5. Add a test:
   pm.test("User has name", function() {
     pm.expect(pm.response.json().name).to.exist;
   });
```

## Insomnia - The Lightweight Alternative

### What Problem It Solves

**Insomnia is a cleaner, lighter alternative to Postman** focused on core API testing without extra features.

**Use cases:**
- Manual API testing
- Quick request prototyping
- GraphQL API testing
- When Postman feels too heavy

### Key Features

#### 1. Clean, Minimal Interface
```
Simple, distraction-free UI:
- Less cluttered than Postman
- Faster startup
- Cleaner workspace organization
```

#### 2. Native GraphQL Support
```graphql
# Better GraphQL experience than Postman
query GetUser($id: ID!) {
  user(id: $id) {
    name
    email
    posts {
      title
    }
  }
}

Variables:
{
  "id": "123"
}

# Auto-completion, schema introspection built-in
```

#### 3. Template Tags (Dynamic Values)
```javascript
// Similar to Postman variables but inline
GET https://api.example.com/users/{% userId %}

Authorization: Bearer {% token %}

// Press Ctrl+Space to see available templates:
- Now (current timestamp)
- UUID (random UUID)
- Base64 encode/decode
- Response values from other requests
```

#### 4. Environment Switching
```json
// Base Environment
{
  "base_url": "https://api.example.com",
  "api_version": "v1"
}

// Development (inherits base)
{
  "base_url": "http://localhost:3000"
}

// Production (inherits base)
{
  "api_key": "prod_key_xyz"
}

// Use: {{ base_url }}/{{ api_version }}/users
```

#### 5. Code Snippet Generation
```javascript
// Generate request code for any language
// HTTP, JavaScript, Python, Go, Swift, etc.

// Example - generates this from your request:
fetch('https://api.example.com/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer abc123'
  },
  body: JSON.stringify({
    name: 'John Doe'
  })
});
```

### Pros and Cons

#### ✅ Advantages

1. **Lightweight** - Much faster and lighter than Postman (200MB vs 500MB RAM)
2. **Clean UI** - Less cluttered, easier to focus
3. **Better GraphQL support** - First-class GraphQL tooling
4. **Open source** - Core is open source (unlike Postman)
5. **No forced account** - Can use without login
6. **Fast startup** - Opens quickly
7. **Cross-platform** - Windows, Mac, Linux
8. **Plugin system** - Extensible with plugins

#### ❌ Disadvantages

1. **Fewer features** - No mock servers, monitoring, etc.
2. **Smaller community** - Less resources/tutorials
3. **Limited collaboration** - Team features not as robust
4. **No automated testing** - Can't run test suites like Postman
5. **Less mature** - Newer tool, some bugs
6. **Documentation generation limited** - Not as comprehensive as Postman

### When to Choose Insomnia

✅ **Best for:**
- GraphQL APIs (better than Postman)
- Developers who want minimal, fast tool
- Solo developers (don't need team features)
- When Postman feels bloated
- Resource-constrained systems
- Open source preference

❌ **Avoid when:**
- Need comprehensive team collaboration
- Want automated test suites
- Need mock servers
- Require extensive documentation generation

### Quick Start Example

```
1. Download Insomnia from insomnia.rest

2. Create request:
   - Click "+"
   - Select "HTTP Request"
   - URL: https://jsonplaceholder.typicode.com/users/1
   - Hit Ctrl+Enter (or click Send)

3. GraphQL example:
   - Click "+"
   - Select "GraphQL Request"
   - URL: https://countries.trevorblades.com/
   - Query:
     query {
       country(code: "US") {
         name
         capital
       }
     }
   - Ctrl+Enter
```

## curl - The Universal Command-Line Tool

### What Problem It Solves

**curl is a command-line tool for transferring data with URLs** - comes pre-installed on Mac/Linux.

**Use cases:**
- Quick API tests from terminal
- Scripting and automation
- CI/CD pipelines
- Remote server testing (no GUI available)
- Sharing exact requests (one-line commands)
- Testing APIs in production environments

### Key Features

#### 1. Simple GET Request
```bash
# Basic GET request
curl https://api.example.com/users

# Output: JSON response directly to terminal
```

#### 2. Custom Headers
```bash
# Add headers with -H flag
curl https://api.example.com/users \
  -H "Authorization: Bearer abc123" \
  -H "Content-Type: application/json"
```

#### 3. POST Requests with Data
```bash
# POST with JSON data
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{
    "name": "John Doe",
    "email": "john@example.com"
  }'

# POST with data from file
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d @user.json
```

#### 4. Show Response Headers
```bash
# Include response headers in output
curl -i https://api.example.com/users

# Output:
# HTTP/1.1 200 OK
# Content-Type: application/json
# Content-Length: 1234
#
# {"id": 1, "name": "John"}

# Show only headers (like HTTP HEAD method)
curl -I https://api.example.com/users
```

#### 5. Verbose Output (Debugging)
```bash
# Show full request/response details
curl -v https://api.example.com/users

# Output shows:
# * Connected to api.example.com
# > GET /users HTTP/1.1
# > Host: api.example.com
# > User-Agent: curl/7.79.1
# < HTTP/1.1 200 OK
# < Content-Type: application/json
# ... full details
```

#### 6. Save Response to File
```bash
# Save to file
curl https://api.example.com/users -o users.json

# Show progress, save to file
curl https://api.example.com/large-file.zip -o file.zip --progress-bar
```

#### 7. Follow Redirects
```bash
# Follow 301/302 redirects automatically
curl -L https://api.example.com/old-endpoint

# Without -L, stops at redirect
```

#### 8. Authentication
```bash
# Basic authentication
curl -u username:password https://api.example.com/users

# Bearer token
curl https://api.example.com/users \
  -H "Authorization: Bearer abc123"

# API key in query
curl "https://api.example.com/users?api_key=abc123"
```

#### 9. Different HTTP Methods
```bash
# PUT
curl -X PUT https://api.example.com/users/1 \
  -H "Content-Type: application/json" \
  -d '{"name": "Updated Name"}'

# PATCH
curl -X PATCH https://api.example.com/users/1 \
  -d '{"email": "new@example.com"}'

# DELETE
curl -X DELETE https://api.example.com/users/1

# OPTIONS (check allowed methods)
curl -X OPTIONS https://api.example.com/users
```

#### 10. Testing in Scripts
```bash
#!/bin/bash
# Test API health in script

response=$(curl -s -w "%{http_code}" https://api.example.com/health)
http_code="${response: -3}"

if [ "$http_code" == "200" ]; then
  echo "API is healthy"
else
  echo "API is down! Status: $http_code"
  exit 1
fi
```

### Pros and Cons

#### ✅ Advantages

1. **Pre-installed** - Available on Mac/Linux by default
2. **Lightweight** - Minimal resource usage
3. **Scriptable** - Easy to use in automation
4. **Universal** - Works everywhere (servers, containers, CI/CD)
5. **No dependencies** - Just the binary
6. **Fast** - Instant, no startup time
7. **Shareable** - One command shares exact request
8. **Documentation** - Extensive man pages
9. **Powerful** - Supports everything (HTTP/2, FTP, etc.)

#### ❌ Disadvantages

1. **Command-line only** - No GUI
2. **Complex syntax** - Flags can be confusing
3. **Hard to read** - Long commands get messy
4. **No request history** - Have to re-type or save commands
5. **Learning curve** - Need to remember flags
6. **No collections** - Can't organize requests easily
7. **Limited debugging** - Harder to inspect than GUI tools

### When to Choose curl

✅ **Best for:**
- Quick one-off requests
- Scripting and automation
- CI/CD pipelines
- Remote servers (SSH access, no GUI)
- Sharing exact requests (paste one command)
- Minimal environments (Docker, cloud instances)
- When GUI tools aren't available

❌ **Avoid when:**
- Building complex test suites
- Need to save/organize many requests
- Prefer visual tools
- Working with team (hard to share curl commands)

### Common curl Patterns

```bash
# Test if API is responding
curl -I https://api.example.com/health

# Time the request
curl -w "@curl-format.txt" -o /dev/null -s https://api.example.com/users
# curl-format.txt contains: time_total: %{time_total}s

# Test with different User-Agent
curl -A "MyApp/1.0" https://api.example.com/users

# Ignore SSL certificate errors (testing only!)
curl -k https://self-signed.example.com/api

# Upload file
curl -X POST https://api.example.com/upload \
  -F "file=@document.pdf" \
  -F "description=My document"

# Pretty-print JSON response (with jq)
curl https://api.example.com/users | jq .

# Test with timeout
curl --max-time 5 https://slow-api.example.com/users
```

## HTTPie - Modern curl Alternative

### What Problem It Solves

**HTTPie makes command-line API testing human-friendly** with intuitive syntax and colored output.

**Use cases:**
- Command-line API testing with better UX than curl
- Quick debugging from terminal
- Scripting with readable commands
- Learning APIs interactively

### Key Features

#### 1. Simple, Intuitive Syntax
```bash
# HTTPie - Clean and readable
http GET https://api.example.com/users

# vs curl (more verbose)
curl https://api.example.com/users
```

#### 2. Automatic JSON
```bash
# HTTPie - assumes JSON, auto-formats
http POST https://api.example.com/users name="John" email="john@example.com"

# vs curl (manual JSON)
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name":"John","email":"john@example.com"}'
```

**Explanation:** HTTPie automatically sets `Content-Type: application/json` and formats data.

#### 3. Colored Output
```bash
# HTTPie - syntax-highlighted, colored
http https://api.example.com/users

# Output:
# HTTP/1.1 200 OK          (colored blue)
# Content-Type: application/json (colored green)
#
# {
#   "name": "John",        (colored, indented)
#   "id": 1
# }
```

**Explanation:** Much easier to read than plain curl output.

#### 4. Readable Headers
```bash
# HTTPie - simple header syntax
http https://api.example.com/users \
  Authorization:"Bearer abc123" \
  X-Custom-Header:"value"

# vs curl
curl https://api.example.com/users \
  -H "Authorization: Bearer abc123" \
  -H "X-Custom-Header: value"
```

#### 5. Form Data
```bash
# HTTPie - form data
http -f POST https://api.example.com/login \
  username=john \
  password=secret

# Automatically sets: Content-Type: application/x-www-form-urlencoded
```

#### 6. File Upload
```bash
# HTTPie - upload file
http -f POST https://api.example.com/upload \
  file@document.pdf \
  description="My document"

# Clean syntax compared to curl's -F flag
```

#### 7. Sessions (Save Auth)
```bash
# Create session (saves cookies, auth)
http --session=./session.json POST https://api.example.com/login \
  username=john password=secret

# Reuse session (automatically includes saved auth)
http --session=./session.json GET https://api.example.com/profile

# No need to pass auth token again!
```

#### 8. Download Files
```bash
# Download with progress bar
http --download https://api.example.com/file.zip

# Save with custom name
http --download --output=myfile.zip https://api.example.com/file.zip
```

#### 9. Request Items Types
```bash
# Different data types (HTTPie recognizes patterns)
http POST https://api.example.com/data \
  name=John \              # String (default)
  age:=30 \                # Raw JSON (number)
  active:=true \           # Boolean
  tags:='["a","b"]' \      # JSON array
  metadata:='{"key":"val"}' # JSON object

# Generates:
# {
#   "name": "John",
#   "age": 30,
#   "active": true,
#   "tags": ["a", "b"],
#   "metadata": {"key": "val"}
# }
```

#### 10. Debugging
```bash
# Show request and response headers
http -v GET https://api.example.com/users

# Show only headers
http -h GET https://api.example.com/users

# Show only body
http -b GET https://api.example.com/users
```

### Pros and Cons

#### ✅ Advantages

1. **Intuitive syntax** - Much easier than curl
2. **Colored output** - Easy to read
3. **Auto JSON** - Assumes JSON by default
4. **Session support** - Save auth between requests
5. **Pretty formatting** - Indented, highlighted output
6. **Better error messages** - More helpful than curl
7. **Great for learning** - Clear syntax helps understand HTTP
8. **Plugins** - Extensible

#### ❌ Disadvantages

1. **Not pre-installed** - Need to install (unlike curl)
2. **Less universal** - curl more common on servers
3. **Fewer features** - curl more powerful for edge cases
4. **Opinionated** - Assumes JSON (sometimes unwanted)
5. **Different syntax** - Have to learn new commands

### When to Choose HTTPie

✅ **Best for:**
- Interactive API exploration from terminal
- Learning HTTP/REST concepts
- Quick debugging with readable output
- When curl syntax feels too complex
- Local development testing

❌ **Avoid when:**
- Need maximum compatibility (use curl)
- Working on servers where installation is restricted
- In CI/CD (curl more standard)
- Need non-HTTP protocols (FTP, etc.)

### Installation

```bash
# Ubuntu/Debian

# Python (any OS)

# Windows
choco install httpie
```

### Quick Start Example

```bash

http https://jsonplaceholder.typicode.com/users/1
```

## Practical Workflow: Using Tools Together

Most developers use **multiple tools** for different purposes:

### Typical Developer Workflow

```
Morning: Exploring new API
├─ HTTPie from terminal
│  └─ Quick, interactive exploration
│
During Development: Building features  
├─ Postman for manual testing
│  └─ Save requests to collection
│  └─ Share with team
│
Before Commit: Automated checks
├─ curl in pre-commit hook
│  └─ Verify API is responding
│
CI/CD Pipeline: 
├─ curl in deployment script
│  └─ Health check after deploy
│
Production Debugging:
├─ curl on production server
   └─ SSH in, test directly
```

### Example: Full Development Cycle

```bash
# 1. Explore API with HTTPie (easy to read)
http https://api.example.com/users

# 2. Save complex request to Postman collection
#    Visual interface for team sharing

# 3. Generate curl command from Postman
#    Add to deployment script:
curl -f https://api.example.com/health || exit 1

# 4. Production issue? SSH and use curl
ssh production-server
curl -v https://localhost:3000/users/123
```

## Key Takeaways

### When Building APIs:
1. **Start with HTTPie/curl** - Quick testing during development
2. **Build Postman collection** - Organize requests, share with team
3. **Use curl in CI/CD** - Automated health checks
4. **Keep Postman updated** - Living documentation

### General Principles:
- **GUI tools (Postman/Insomnia)** - Best for exploration, organization, collaboration
- **CLI tools (curl/HTTPie)** - Best for automation, scripting, quick tests
- **Don't limit yourself to one** - Use the right tool for each task
- **Learn curl** - Universal, works everywhere
- **Use Postman for teams** - Collaboration features worth it

### Remember:
- All tools solve the same core problem (testing HTTP requests)
- Choose based on context (team size, workflow, environment)
- GUI tools are easier to learn, CLI tools are more powerful for automation
- The best tool is the one you're comfortable with and fits your workflow

No single tool is perfect - professional developers typically use 2-3 of these tools depending on the task at hand.
