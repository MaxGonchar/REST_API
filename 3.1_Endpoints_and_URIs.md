# Endpoints and URIs in REST API

## Introduction

An **endpoint** is a specific URL where an API can be accessed. The **URI** (Uniform Resource Identifier) uniquely identifies a resource. Together, they form the foundation of how clients interact with your REST API.

**Key principle:** The URI identifies **what** (the resource), and the HTTP method specifies **what to do** with it.

```
https://api.example.com/v1/users/42
│      │               │  │     │
│      │               │  │     └─ Resource identifier
│      │               │  └─────── Resource type
│      │               └────────── API version
│      └────────────────────────── Domain
└───────────────────────────────── Protocol (always HTTPS)
```

---

## URI Anatomy

### Complete URI Structure

```
https://api.example.com:443/v1/users/42/orders?status=pending&sort=date#section
│      │               │   │  │     │  │      │                          │
│      │               │   │  │     │  │      └─ Query parameters        └─ Fragment
│      │               │   │  │     │  └─ Nested resource
│      │               │   │  │     └─ Resource ID
│      │               │   │  └─ Resource collection
│      │               │   └─ API version
│      │               └─ Port (optional, 443 is default for HTTPS)
│      └─ Domain/Host
└─ Scheme (protocol)
```

### Components Breakdown

**1. Scheme (Protocol)**
```
https://  ← Always use HTTPS for REST APIs
```

**2. Host/Domain**
```
api.example.com
api-staging.example.com
subdomain.api.example.com
```

**3. Port (optional)**
```
:443   (default for HTTPS, usually omitted)
:8080  (custom port)
```

**4. Base Path**
```
/api
/api/v1
/v1
/
```

**5. Resource Path**
```
/users                  (collection)
/users/42               (specific resource)
/users/42/orders        (nested resource)
/users/42/orders/789    (specific nested resource)
```

**6. Query Parameters**
```
?status=active&sort=name&limit=20
```

**7. Fragment (NOT used in REST APIs)**
```
#section-2
```

**Important:** Fragments are client-side only and **never sent to the server**. They're processed by the browser for scrolling to page sections or client-side routing (SPAs). Not applicable to REST API design.

## Base URL Structure

### Simple Base URL

```
https://api.example.com/
```

**Endpoints:**
```
GET    https://api.example.com/users
GET    https://api.example.com/products
POST   https://api.example.com/orders
```

### Base URL with Version

```
https://api.example.com/v1/
```

**Endpoints:**
```
GET    https://api.example.com/v1/users
GET    https://api.example.com/v2/users  (newer version)
```

### Base URL with API Path

```
https://example.com/api/v1/
```

**Endpoints:**
```
GET    https://example.com/api/v1/users
POST   https://example.com/api/v1/users
```

### Subdomain for API

```
https://api.example.com/v1/
```

**Benefits:**
- Separate API infrastructure
- Easier scaling
- Better security isolation
- Cleaner DNS management

### Environment-Specific Base URLs

```
Production:  https://api.example.com/v1/
Staging:     https://api-staging.example.com/v1/
Development: https://api-dev.example.com/v1/
Local:       http://localhost:3000/api/v1/
```

## Resource Paths

### Basic Resource Patterns

#### Collections (Plural Nouns)

```
GET    /users           → List all users
POST   /users           → Create new user
GET    /products        → List all products
POST   /products        → Create new product
```

**Why plural?** 
```
/users/42    ← Makes sense: "user 42 from users collection"
/user/42     ← Less clear: "user 42 from user collection"
```

#### Specific Resources

```
GET    /users/42        → Get user 42
PUT    /users/42        → Update user 42
PATCH  /users/42        → Partially update user 42
DELETE /users/42        → Delete user 42
```

#### Nested Resources

```
GET    /users/42/orders           → Orders belonging to user 42
POST   /users/42/orders           → Create order for user 42
GET    /users/42/orders/789       → Specific order of user 42
PUT    /users/42/orders/789       → Update order 789 of user 42
DELETE /users/42/orders/789       → Delete order 789 of user 42
```

## Path Parameters vs Query Parameters

### Path Parameters

**Use for:** Resource identification and hierarchy

```
/users/{userId}
/users/{userId}/orders/{orderId}
/categories/{categoryId}/products/{productId}
```

**Examples:**
```
GET /users/42                    → Get user with ID 42
GET /users/42/orders/789         → Get order 789 of user 42
GET /posts/2024-01-15/comments   → Comments on specific date
```

**Characteristics:**
- ✅ Required (not optional)
- ✅ Part of resource identity
- ✅ SEO-friendly
- ✅ Clear hierarchy

### Query Parameters

**Use for:** Filtering, sorting, pagination, optional parameters

```
/users?status=active&role=admin&sort=name
```

**Examples:**

#### Filtering
```
GET /users?status=active
GET /products?category=electronics&minPrice=100&maxPrice=500
GET /orders?startDate=2024-01-01&endDate=2024-12-31
```

#### Sorting
```
GET /users?sort=name
GET /users?sort=name&order=asc
GET /products?sort=price&order=desc
GET /posts?sort=createdAt&order=desc
```

#### Pagination
```
GET /users?page=2&limit=20
GET /users?offset=40&limit=20
GET /posts?cursor=abc123
```

#### Field Selection
```
GET /users?fields=id,name,email
GET /products?fields=id,name,price
```

#### Search
```
GET /users?search=john
GET /products?q=laptop
GET /articles?query=rest+api
```

#### Multiple Values
```
GET /products?tags=electronics&tags=sale
GET /users?id=1&id=2&id=3
GET /posts?status=draft&status=published
```

**Characteristics:**
- ✅ Optional (usually)
- ✅ Modify response
- ✅ Not part of resource identity
- ✅ Can be combined

### Decision Matrix

| Use Case | Path Parameter | Query Parameter |
|----------|---------------|-----------------|
| Resource ID | ✅ `/users/42` | ❌ |
| Filtering | ❌ | ✅ `?status=active` |
| Sorting | ❌ | ✅ `?sort=name` |
| Pagination | ❌ | ✅ `?page=2` |
| Optional param | ❌ | ✅ `?include=profile` |
| Resource hierarchy | ✅ `/users/42/orders` | ❌ |

## Naming Conventions

### 1. Use Nouns, Not Verbs

```
✅ Good:
GET    /users              (retrieve users)
POST   /orders             (create order)
DELETE /products/42        (delete product)

❌ Bad:
GET    /getUsers
POST   /createOrder
GET    /deleteProduct/42
```

**Exception:** Actions/operations (covered later)

### 2. Use Plural Forms

```
✅ Good:
/users
/products
/orders
/categories

❌ Bad:
/user
/product
/order
/category
```

**Exception:** Singleton resources (see special cases)

### 3. Use Lowercase with Hyphens

```
✅ Good:
/user-profiles
/order-items
/shopping-carts
/payment-methods

❌ Avoid:
/UserProfiles           (PascalCase)
/userProfiles           (camelCase)
/user_profiles          (snake_case)
/SHOPPING-CARTS         (UPPERCASE)
```

**Why hyphens?**
- URLs are case-insensitive in some systems
- Better readability
- SEO-friendly
- Standard practice

### 4. Keep It Simple and Intuitive

```
✅ Good:
/users/42/orders
/products/123/reviews
/posts/456/comments

❌ Too complex:
/api/v1/management/system/users/data/42/personal/orders/history
```

### 5. Avoid Deep Nesting

```
✅ Good:
/users/42/orders
/reviews?productId=123

❌ Bad:
/users/42/accounts/5/orders/789/items/3/reviews/99
```

**Rule of thumb:** Max 2-3 levels of nesting

## Special Cases and Advanced Patterns

### Case 1: POST for Complex Search

**Problem:** GET with complex search criteria becomes unwieldy or hits URL length limits.

**URL length limits:**
- Browsers: ~2000 characters
- Some proxies: 1024 characters
- Servers: Configurable but often 8KB

#### When GET is problematic:

```javascript
// Complex search with many filters
GET /products?
    category=electronics&
    subcategory=laptops&
    brands=apple&brands=dell&brands=hp&
    minPrice=500&maxPrice=2000&
    features=ssd&features=touchscreen&features=backlit&
    ram=8GB&ram=16GB&
    screen=13inch&screen=15inch&
    conditions=new&conditions=refurbished&
    sellers=amazon&sellers=bestbuy&
    ratings=4&ratings=5&
    shipping=free&
    availability=in-stock&
    sortBy=price&
    order=asc&
    page=1&
    limit=20

// This URL is 400+ characters and hard to manage
```

#### Solution: POST for search

```javascript
POST /products/search
Content-Type: application/json

{
  "filters": {
    "category": "electronics",
    "subcategory": "laptops",
    "brands": ["apple", "dell", "hp"],
    "priceRange": {
      "min": 500,
      "max": 2000
    },
    "features": ["ssd", "touchscreen", "backlit"],
    "specs": {
      "ram": ["8GB", "16GB"],
      "screen": ["13inch", "15inch"]
    },
    "conditions": ["new", "refurbished"],
    "sellers": ["amazon", "bestbuy"],
    "ratings": [4, 5],
    "shipping": "free",
    "availability": "in-stock"
  },
  "sort": {
    "field": "price",
    "order": "asc"
  },
  "pagination": {
    "page": 1,
    "limit": 20
  }
}
```

**Response:**
```json
{
  "results": [
    { "id": 1, "name": "MacBook Pro", "price": 1299 },
    { "id": 2, "name": "Dell XPS", "price": 1199 }
  ],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 150,
    "pages": 8
  }
}
```

**Benefits:**
- No URL length limits
- Complex nested structures
- Better readability
- Reusable (save search criteria)

**Alternative endpoint patterns:**
```
POST /products/search
POST /search/products
POST /products:search     (Google API style)
POST /_search/products    (Elasticsearch style)
```

**Implementation note:**
```javascript
// Make it clear this is a read operation
app.post('/products/search', (req, res) => {
  const { filters, sort, pagination } = req.body;
  
  // Important: This should NOT modify data
  const results = searchProducts(filters, sort, pagination);
  
  res.json({
    results,
    pagination
  });
});
```

### Case 4: Relationships Between Resources

#### Pattern 1: Nested Resources (Strong ownership)

```javascript
// Comments belong to posts
GET    /posts/42/comments
POST   /posts/42/comments
GET    /posts/42/comments/123
DELETE /posts/42/comments/123

// Orders belong to users
GET    /users/42/orders
POST   /users/42/orders
```

#### Pattern 2: Query Parameters (Weak relationship)

```javascript
// Get comments by post (alternative to nesting)
GET /comments?postId=42

// Get orders by user
GET /orders?userId=42

// Get products by category
GET /products?categoryId=5
```

#### Pattern 3: Relationship as Resource

```javascript
// Many-to-many: Users and Groups

// Add user to group
POST /users/42/memberships
{
  "groupId": 10
}

// Remove user from group
DELETE /users/42/memberships/membership-id-123

// Get user's groups
GET /users/42/memberships

// Alternative: from group perspective
GET /groups/10/members
POST /groups/10/members
{
  "userId": 42,
  "role": "moderator"
}
```

## Best Practices Summary

### 1. Keep URIs Clean and Predictable

```javascript
✅ Good:
/users
/users/42
/users/42/orders
/orders?userId=42

❌ Bad:
/getUsers
/user_42
/user/42/order_list
/getOrdersByUser?id=42
```

### 2. Use Appropriate HTTP Methods

```javascript
GET    /users/42        (retrieve)
POST   /users           (create)
PUT    /users/42        (update/replace)
PATCH  /users/42        (partial update)
DELETE /users/42        (delete)
POST   /users/42/activate (action)
```

### 3. Limit Nesting Depth

```javascript
✅ Good:
/users/42/orders
/orders?userId=42

❌ Too deep:
/users/42/accounts/5/orders/789/items/3
```

### 4. Use Query Parameters for Filtering

```javascript
✅ Good:
/products?category=electronics&status=active

❌ Bad:
/products/electronics/active
```

### 5. Version Your API

```javascript
✅ Good:
/v1/users
/v2/users

❌ Risky:
/users  (no versioning)
```

### 6. Use Standard Formats for Special Values

```javascript
// Dates: ISO 8601
?date=2026-01-13
?startDate=2026-01-01T00:00:00Z

// Booleans
?active=true
?deleted=false

// Lists
?tags=tag1,tag2,tag3
?ids=1&ids=2&ids=3
```

### 7. Document Your Endpoints

```yaml
# OpenAPI/Swagger
/users/{userId}:
  get:
    summary: Get user by ID
    parameters:
      - name: userId
        in: path
        required: true
        schema:
          type: integer
    responses:
      200:
        description: User found
      404:
        description: User not found
```

## Common Mistakes to Avoid

### 1. Verbs in URIs

```javascript
❌ /getUsers
❌ /createOrder
❌ /deleteProduct

✅ GET /users
✅ POST /orders
✅ DELETE /products/{id}
```

### 2. Actions That Should Be Resources

```javascript
❌ POST /addToCart
❌ POST /removeFromCart

✅ POST   /cart/items
✅ DELETE /cart/items/{id}
```

### 3. Mixing Singular and Plural

```javascript
❌ /user/{id}
❌ /products
   /product/{id}

✅ /users/{id}
✅ /products
   /products/{id}
```

### 4. CamelCase or snake_case

```javascript
❌ /userProfiles
❌ /user_profiles

✅ /user-profiles
```

### 5. Including File Extensions

```javascript
❌ /users.json
❌ /products.xml

✅ /users
   Accept: application/json
```

## Summary

**URI Design Principles:**
1. **Resources are nouns** - `/users`, `/products`, `/orders`
2. **Actions are HTTP methods** - GET, POST, PUT, DELETE
3. **Hierarchy shows relationships** - `/users/42/orders`
4. **Query params filter/modify** - `?status=active&sort=name`
5. **Keep it simple** - 2-3 levels max nesting
6. **Be consistent** - Follow patterns throughout API

**Special Cases:**
- POST for complex search - When GET is insufficient
- Actions on resources - `/resource/{id}/action`
- Controller resources - `/export`, `/import`, `/search`
- Singleton resources - `/profile`, `/cart`, `/settings`
- Batch operations - `/bulk-update`, `/bulk-delete`

**Key Takeaway:** Good URI design makes your API intuitive, predictable, and easy to use.
