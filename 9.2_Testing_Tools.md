# Testing Tools - Automated API Testing

## The Problem: Why Manual Testing Isn't Enough

When developing APIs, manual testing with Postman/curl works for exploration, but it has critical limitations:

### Problems with Manual Testing

❌ **Slow and repetitive**
```
After every code change:
1. Open Postman
2. Click "Send" on 20+ requests
3. Manually verify each response
4. Check status codes, headers, body
5. Takes 10-15 minutes per test cycle
```

❌ **Human error**
```
Day 1: Test passes ✓
Day 2: Forget to test edge case
Day 3: Bug ships to production 💥
```

❌ **No regression detection**
```
You fix endpoint A
Unknowingly break endpoint B
Manual testing didn't catch it
Users report the bug
```

❌ **Can't run in CI/CD**
```
Deploy to production:
❌ Can't automatically verify API works
❌ Can't prevent broken deployments
❌ Can't run tests on every commit
```

### The Solution: Automated Testing

**Automated tests** run instantly, catch regressions, and integrate with CI/CD pipelines.

```javascript
// One command runs ALL tests in seconds
npm test

✓ GET /users returns 200
✓ GET /users returns array
✓ POST /users creates user
✓ PUT /users/:id updates user
✓ DELETE /users/:id removes user
✓ Invalid auth returns 401
... 50 more tests

Tests: 55 passed
Time: 2.3s
```

## Test Types Overview

### 1. Unit Tests
**Test individual endpoint logic in isolation**

```javascript
// Test function logic without HTTP requests
describe('validateEmail', () => {
  it('accepts valid email', () => {
    expect(validateEmail('test@example.com')).toBe(true);
  });
  
  it('rejects invalid email', () => {
    expect(validateEmail('invalid')).toBe(false);
  });
});
```

**Tools:** Jest, Mocha, pytest (general-purpose test frameworks)

### 2. Integration Tests
**Test full HTTP request/response cycle**

```javascript
// Actually make HTTP request to API
it('POST /users creates user', async () => {
  const response = await request(app)
    .post('/users')
    .send({ name: 'John', email: 'john@example.com' })
    .expect(201);
  
  expect(response.body.name).toBe('John');
});
```

**Tools:** Supertest (Node.js), REST Assured (Java)

### 3. Load Tests
**Test API performance under heavy load**
- How many requests/sec can handle?
- Response time under load?
- When does it crash?

**Tools:** Artillery, k6, Apache JMeter (covered in another topic)

## Jest - JavaScript Testing Framework

### What Problem It Solves

**Jest is a complete JavaScript testing framework** with built-in assertions, mocking, and coverage reporting.

**Use cases:**
- Testing Node.js APIs (with Supertest)
- Testing API logic (validation, business rules)
- Testing JavaScript/TypeScript code
- Running automated test suites

### Key Features

#### 1. Simple Test Syntax
```javascript
// test/user.test.js
describe('User API', () => {
  test('should validate email format', () => {
    expect(isValidEmail('test@example.com')).toBe(true);
    expect(isValidEmail('invalid')).toBe(false);
  });
  
  test('should hash passwords', () => {
    const hashed = hashPassword('secret123');
    expect(hashed).not.toBe('secret123'); // Password is hashed
    expect(hashed.length).toBeGreaterThan(20);
  });
});
```

**Explanation:** `describe` groups related tests, `test`/`it` defines individual tests, `expect` makes assertions.

#### 2. Async/Await Support
```javascript
// Testing async functions
test('should fetch user from database', async () => {
  const user = await User.findById(123);
  expect(user.name).toBe('John Doe');
});

// Testing promises
test('API returns data', () => {
  return fetchUser(1).then(user => {
    expect(user.id).toBe(1);
  });
});
```

#### 3. Built-in Mocking
```javascript
// Mock database to avoid real DB calls
jest.mock('../models/User');

test('creates user without hitting database', async () => {
  // Mock User.create to return fake user
  User.create.mockResolvedValue({
    id: 1,
    name: 'John',
    email: 'john@example.com'
  });
  
  const user = await createUser({ name: 'John', email: 'john@example.com' });
  
  expect(user.id).toBe(1);
  expect(User.create).toHaveBeenCalledWith({ name: 'John', email: 'john@example.com' });
});
```

**Explanation:** Mocking lets you test logic without external dependencies (database, APIs).

#### 4. Snapshot Testing
```javascript
// Test API response structure
test('user object has correct structure', async () => {
  const user = await getUser(1);
  
  // Creates snapshot file on first run
  // Future runs compare against snapshot
  expect(user).toMatchSnapshot();
});

// Snapshot file (auto-generated):
// exports[`user object has correct structure 1`] = `
// {
//   "id": 1,
//   "name": "John Doe",
//   "email": "john@example.com",
//   "createdAt": "2026-01-15T10:30:00Z"
// }
// `;
```

**Explanation:** Snapshots detect unintended changes to API responses.

#### 5. Code Coverage
```bash
# Run tests with coverage report
npm test -- --coverage

# Output:
# -------------------|---------|----------|---------|---------|
# File               | % Stmts | % Branch | % Funcs | % Lines |
# -------------------|---------|----------|---------|---------|
# users.js           |   95.5  |   88.2   |  100    |   95.5  |
# auth.js            |   78.3  |   66.7   |   85.7  |   78.3  |
# -------------------|---------|----------|---------|---------|
```

**Explanation:** Shows which code is tested and which isn't.

#### 6. Watch Mode
```bash
# Re-run tests automatically when files change
npm test -- --watch

# Only runs tests related to changed files
# Instant feedback while developing
```

### Complete Example: Testing API with Jest + Supertest

```javascript
// test/api/users.test.js
const request = require('supertest');
const app = require('../../app');
const User = require('../../models/User');

// Setup & teardown
beforeAll(async () => {
  // Connect to test database
  await connectTestDB();
});

afterAll(async () => {
  // Clean up
  await closeDB();
});

beforeEach(async () => {
  // Clear database before each test
  await User.deleteMany({});
});

describe('GET /api/users', () => {
  test('returns empty array when no users', async () => {
    const response = await request(app)
      .get('/api/users')
      .expect(200);
    
    expect(response.body).toEqual([]);
  });
  
  test('returns all users', async () => {
    // Create test data
    await User.create([
      { name: 'John', email: 'john@example.com' },
      { name: 'Jane', email: 'jane@example.com' }
    ]);
    
    const response = await request(app)
      .get('/api/users')
      .expect(200);
    
    expect(response.body).toHaveLength(2);
    expect(response.body[0].name).toBe('John');
  });
});

describe('POST /api/users', () => {
  test('creates new user', async () => {
    const userData = {
      name: 'John Doe',
      email: 'john@example.com'
    };
    
    const response = await request(app)
      .post('/api/users')
      .send(userData)
      .expect(201);
    
    expect(response.body).toMatchObject(userData);
    expect(response.body.id).toBeDefined();
    
    // Verify in database
    const user = await User.findById(response.body.id);
    expect(user.name).toBe('John Doe');
  });
  
  test('validates required fields', async () => {
    const response = await request(app)
      .post('/api/users')
      .send({ name: 'John' }) // Missing email
      .expect(400);
    
    expect(response.body.error).toContain('email');
  });
  
  test('validates email format', async () => {
    const response = await request(app)
      .post('/api/users')
      .send({ name: 'John', email: 'invalid-email' })
      .expect(400);
    
    expect(response.body.error).toContain('valid email');
  });
});

describe('Authentication', () => {
  test('requires auth token', async () => {
    await request(app)
      .get('/api/profile')
      .expect(401);
  });
  
  test('accepts valid token', async () => {
    const user = await User.create({ name: 'John', email: 'john@example.com' });
    const token = generateToken(user);
    
    const response = await request(app)
      .get('/api/profile')
      .set('Authorization', `Bearer ${token}`)
      .expect(200);
    
    expect(response.body.email).toBe('john@example.com');
  });
});
```

### Pros and Cons

#### ✅ Advantages

1. **Zero configuration** - Works out of the box
2. **Fast** - Parallel test execution
3. **Built-in everything** - Assertions, mocking, coverage
4. **Great error messages** - Clear, helpful output
5. **Snapshot testing** - Easy to test complex objects
6. **Watch mode** - Instant feedback
7. **Large community** - Tons of resources
8. **Works with TypeScript** - Built-in support

#### ❌ Disadvantages

1. **JavaScript/TypeScript only** - Can't test other languages
2. **Can be slow** - On very large test suites
3. **Opinionated** - Less flexible than Mocha
4. **Globals** - Uses global functions (describe, test, expect)
5. **Snapshot bloat** - Snapshots can become unwieldy

### When to Choose Jest

✅ **Best for:**
- Node.js API testing
- JavaScript/TypeScript projects
- Teams wanting batteries-included framework
- Projects needing code coverage
- React/Vue apps with APIs
- When you want quick setup

❌ **Avoid when:**
- Testing non-JavaScript APIs
- Need extreme flexibility (use Mocha)
- Very large test suites (10,000+ tests)

```bash
  }
```

## Mocha - Flexible Testing Framework

### What Problem It Solves

**Mocha is a minimalist, flexible testing framework** that lets you choose your own assertion library and tools.

**Use cases:**
- Testing Node.js APIs (with Chai + Supertest)
- When you need maximum flexibility
- Custom test setups
- Projects with specific requirements

### Key Features

#### 1. Flexible - Choose Your Tools
```javascript
// Mocha doesn't include assertions - you choose
const { expect } = require('chai');  // Assertion library
const should = require('should');    // Alternative
const assert = require('assert');    // Node.js built-in

// Use whatever you prefer
describe('User API', () => {
  it('validates email with chai', () => {
    expect(isValidEmail('test@example.com')).to.be.true;
  });
  
  it('validates email with should', () => {
    isValidEmail('test@example.com').should.be.true();
  });
  
  it('validates email with assert', () => {
    assert.strictEqual(isValidEmail('test@example.com'), true);
  });
});
```

**Explanation:** Mocha provides test structure, you provide assertions. More flexible than Jest.

#### 2. Multiple Assertion Styles
```javascript
const chai = require('chai');
const expect = chai.expect;  // BDD style
const assert = chai.assert;  // TDD style

// BDD style (behavior-driven)
expect(user.name).to.equal('John');
expect(user.age).to.be.above(18);
expect(user.email).to.match(/.*@.*/);

// TDD style (test-driven)
assert.equal(user.name, 'John');
assert.isAbove(user.age, 18);
assert.match(user.email, /.*@.*/);
```

#### 3. Hooks for Setup/Teardown
```javascript
describe('User API Tests', () => {
  before(() => {
    // Runs once before all tests
    console.log('Starting test suite');
  });
  
  after(() => {
    // Runs once after all tests
    console.log('Finished test suite');
  });
  
  beforeEach(async () => {
    // Runs before each test
    await clearDatabase();
  });
  
  afterEach(() => {
    // Runs after each test
    console.log('Test completed');
  });
  
  it('test 1', () => { /* ... */ });
  it('test 2', () => { /* ... */ });
});
```

#### 4. Async Testing
```javascript
// Callback style
it('fetches user (callback)', (done) => {
  fetchUser(1, (err, user) => {
    expect(user.id).to.equal(1);
    done(); // Signal test complete
  });
});

// Promise style
it('fetches user (promise)', () => {
  return fetchUser(1).then(user => {
    expect(user.id).to.equal(1);
  });
});

// Async/await style
it('fetches user (async/await)', async () => {
  const user = await fetchUser(1);
  expect(user.id).to.equal(1);
});
```

#### 5. Timeouts for Slow Tests
```javascript
describe('Slow API tests', () => {
  // Set timeout for entire suite
  this.timeout(5000); // 5 seconds
  
  it('slow database query', async function() {
    this.timeout(10000); // Override for this test
    
    const result = await slowQuery();
    expect(result).to.exist;
  });
});
```

### Complete Example: Mocha + Chai + Supertest

```javascript
// test/users.test.js
const chai = require('chai');
const chaiHttp = require('chai-http');
const app = require('../app');
const User = require('../models/User');

const expect = chai.expect;
chai.use(chaiHttp);

describe('User API', () => {
  beforeEach(async () => {
    await User.deleteMany({});
  });
  
  describe('GET /api/users', () => {
    it('should return empty array', (done) => {
      chai.request(app)
        .get('/api/users')
        .end((err, res) => {
          expect(res).to.have.status(200);
          expect(res.body).to.be.an('array');
          expect(res.body).to.have.lengthOf(0);
          done();
        });
    });
    
    it('should return all users', async () => {
      await User.create([
        { name: 'John', email: 'john@example.com' },
        { name: 'Jane', email: 'jane@example.com' }
      ]);
      
      const res = await chai.request(app).get('/api/users');
      
      expect(res).to.have.status(200);
      expect(res.body).to.have.lengthOf(2);
      expect(res.body[0]).to.have.property('name', 'John');
    });
  });
  
  describe('POST /api/users', () => {
    it('should create new user', async () => {
      const res = await chai.request(app)
        .post('/api/users')
        .send({ name: 'John', email: 'john@example.com' });
      
      expect(res).to.have.status(201);
      expect(res.body).to.include({ name: 'John' });
      expect(res.body).to.have.property('id');
    });
    
    it('should validate email', async () => {
      const res = await chai.request(app)
        .post('/api/users')
        .send({ name: 'John', email: 'invalid' });
      
      expect(res).to.have.status(400);
      expect(res.body.error).to.match(/email/i);
    });
  });
});
```

### Pros and Cons

#### ✅ Advantages

1. **Extremely flexible** - Choose your own tools
2. **Lightweight** - Minimal core, add what you need
3. **Mature** - Battle-tested, stable
4. **Multiple reporters** - Various output formats
5. **Works in browser** - Can test client-side code
6. **No globals required** - Cleaner than Jest
7. **Popular** - Large ecosystem

#### ❌ Disadvantages

1. **More setup** - Need to choose and configure tools
2. **No built-in mocking** - Need Sinon.js
3. **No code coverage** - Need Istanbul/nyc
4. **Slower than Jest** - Runs tests serially by default
5. **More boilerplate** - More code to write

### When to Choose Mocha

✅ **Best for:**
- Teams wanting maximum flexibility
- Projects with custom requirements
- When you want to pick your own tools
- Browser and Node.js testing
- Large, complex projects

❌ **Avoid when:**
- Want quick, zero-config setup (use Jest)
- Team prefers opinionated tools
- Small projects (Jest easier)

## pytest - Python Testing Framework

### What Problem It Solves

**pytest is the leading Python testing framework** for testing Python APIs (Flask, Django, FastAPI).

**Use cases:**
- Testing Python APIs
- Python backend testing
- Data validation testing
- Microservices in Python

### Key Features

#### 1. Simple Assertions
```python
# test_users.py

def test_validate_email():
    # No special assertion library - just use assert
    assert is_valid_email('test@example.com') == True
    assert is_valid_email('invalid') == False

def test_hash_password():
    hashed = hash_password('secret123')
    assert hashed != 'secret123'
    assert len(hashed) > 20
```

**Explanation:** pytest uses plain Python `assert` statements. No need to learn special methods.

#### 2. Fixtures for Setup
```python
import pytest
from app import app, db
from models import User

@pytest.fixture
def client():
    """Create test client"""
    app.config['TESTING'] = True
    with app.test_client() as client:
        yield client

@pytest.fixture
def init_database():
    """Initialize test database"""
    db.create_all()
    yield db
    db.drop_all()

def test_get_users(client, init_database):
    """Test GET /users"""
    response = client.get('/api/users')
    assert response.status_code == 200
    assert response.json == []
```

**Explanation:** Fixtures provide reusable setup code. pytest automatically runs them based on function parameters.

#### 3. Parametrized Tests
```python
import pytest

@pytest.mark.parametrize("email,expected", [
    ("test@example.com", True),
    ("user@domain.co.uk", True),
    ("invalid", False),
    ("@example.com", False),
    ("test@", False),
])
def test_email_validation(email, expected):
    """Test multiple email formats"""
    assert is_valid_email(email) == expected

# Runs 5 separate tests automatically!
```

**Explanation:** One test function generates multiple test cases. Great for testing edge cases.

#### 4. Markers for Test Organization
```python
import pytest

@pytest.mark.slow
def test_large_query():
    """Long-running test"""
    # ... slow test

@pytest.mark.integration
def test_full_workflow():
    """Integration test"""
    # ... integration test

@pytest.mark.skip(reason="Feature not implemented")
def test_future_feature():
    """Not ready yet"""
    pass

# Run only fast tests:
# pytest -m "not slow"

# Run only integration tests:
# pytest -m integration
```

#### 5. API Testing Example (Flask)
```python
# test_api.py
import pytest
from app import app, db
from models import User

@pytest.fixture
def client():
    app.config['TESTING'] = True
    app.config['SQLALCHEMY_DATABASE_URI'] = 'sqlite:///:memory:'
    
    with app.test_client() as client:
        with app.app_context():
            db.create_all()
        yield client
        with app.app_context():
            db.drop_all()

class TestUserAPI:
    """User API test suite"""
    
    def test_get_users_empty(self, client):
        """GET /users returns empty array"""
        response = client.get('/api/users')
        assert response.status_code == 200
        assert response.json == []
    
    def test_create_user(self, client):
        """POST /users creates user"""
        response = client.post('/api/users', json={
            'name': 'John Doe',
            'email': 'john@example.com'
        })
        
        assert response.status_code == 201
        assert response.json['name'] == 'John Doe'
        assert 'id' in response.json
    
    def test_create_user_validation(self, client):
        """POST /users validates input"""
        response = client.post('/api/users', json={
            'name': 'John'
            # Missing email
        })
        
        assert response.status_code == 400
        assert 'email' in response.json['error']
    
    def test_get_user_by_id(self, client):
        """GET /users/:id returns user"""
        # Create user first
        create_response = client.post('/api/users', json={
            'name': 'John',
            'email': 'john@example.com'
        })
        user_id = create_response.json['id']
        
        # Get user
        response = client.get(f'/api/users/{user_id}')
        assert response.status_code == 200
        assert response.json['name'] == 'John'
    
    def test_update_user(self, client):
        """PUT /users/:id updates user"""
        create_response = client.post('/api/users', json={
            'name': 'John',
            'email': 'john@example.com'
        })
        user_id = create_response.json['id']
        
        response = client.put(f'/api/users/{user_id}', json={
            'name': 'Jane Doe'
        })
        
        assert response.status_code == 200
        assert response.json['name'] == 'Jane Doe'
    
    def test_delete_user(self, client):
        """DELETE /users/:id removes user"""
        create_response = client.post('/api/users', json={
            'name': 'John',
            'email': 'john@example.com'
        })
        user_id = create_response.json['id']
        
        response = client.delete(f'/api/users/{user_id}')
        assert response.status_code == 204
        
        # Verify deleted
        response = client.get(f'/api/users/{user_id}')
        assert response.status_code == 404
```

## Supertest - Node.js HTTP Testing

### What Problem It Solves

**Supertest makes HTTP testing in Node.js simple** with a fluent API for requests and assertions.

**Use cases:**
- Testing Express/Koa/Fastify APIs
- Integration testing Node.js servers
- End-to-end API testing

### Key Features

#### 1. Fluent HTTP Assertions
```javascript
const request = require('supertest');
const app = require('../app');

// Chain expectations
await request(app)
  .get('/api/users/1')
  .expect(200)                          // Status code
  .expect('Content-Type', /json/)       // Header
  .expect(res => {
    expect(res.body.name).toBe('John'); // Body
  });
```

#### 2. All HTTP Methods
```javascript
// GET
await request(app).get('/api/users');

// POST
await request(app)
  .post('/api/users')
  .send({ name: 'John', email: 'john@example.com' });

// PUT
await request(app)
  .put('/api/users/1')
  .send({ name: 'Updated' });

// PATCH
await request(app)
  .patch('/api/users/1')
  .send({ email: 'new@example.com' });

// DELETE
await request(app).delete('/api/users/1');
```

#### 3. Header Testing
```javascript
await request(app)
  .get('/api/users')
  .set('Authorization', 'Bearer token123')     // Set request header
  .expect('Content-Type', /json/)              // Assert response header
  .expect('X-RateLimit-Remaining', '99');      // Custom header
```

#### 4. Authentication Testing
```javascript
describe('Protected endpoints', () => {
  let token;
  
  beforeAll(async () => {
    // Login to get token
    const response = await request(app)
      .post('/api/login')
      .send({ email: 'test@example.com', password: 'secret' });
    
    token = response.body.token;
  });
  
  test('requires authentication', async () => {
    await request(app)
      .get('/api/profile')
      .expect(401); // No token = unauthorized
```

## REST Assured - Java HTTP Testing

### What Problem It Solves

**REST Assured makes testing REST APIs in Java simple** with a fluent, BDD-style syntax.

**Use cases:**
- Testing Java APIs (Spring Boot, JAX-RS)
- Testing any REST API from Java
- Java test automation

### Key Features

#### 1. BDD-Style Syntax
```java
import static io.restassured.RestAssured.*;
import static org.hamcrest.Matchers.*;

@Test
public void testGetUsers() {
    given()
        .baseUri("https://api.example.com")
    .when()
        .get("/users")
    .then()
        .statusCode(200)
        .body("size()", greaterThan(0));
}
```

**Explanation:** Reads like English: "Given base URI, when GET /users, then status 200 and body size > 0"

#### 2. Request Specification
```java
@Test
public void testCreateUser() {
    given()
        .contentType("application/json")
        .header("Authorization", "Bearer token123")
        .body("""
            {
                "name": "John Doe",
                "email": "john@example.com"
            }
            """)
    .when()
        .post("/users")
    .then()
        .statusCode(201)
        .body("name", equalTo("John Doe"))
        .body("id", notNullValue());
}
```

#### 3. Response Validation
```java
@Test
public void testUserResponse() {
    get("/users/1")
    .then()
        .statusCode(200)
        .contentType("application/json")
        .header("Content-Type", "application/json")
        .body("id", equalTo(1))
        .body("name", notNullValue())
        .body("email", containsString("@"))
        .body("createdAt", matchesPattern("\\d{4}-\\d{2}-\\d{2}.*"));
}
```

#### 4. JSON Path Extraction
```java
@Test
public void testExtractData() {
    Response response = get("/users/1");
    
    // Extract values
    int id = response.path("id");
    String name = response.path("name");

    String firstHobby = response.path("hobbies[0]");
    
    assertEquals(1, id);
    assertEquals("John Doe", name);
}
```

#### 5. Request/Response Logging
```java

import io.restassured.response.Response;
import org.junit.jupiter.api.*;
import static io.restassured.RestAssured.*;
import static org.hamcrest.Matchers.*;

public class UserAPITest {
    
    @BeforeAll
    public static void setup() {
        RestAssured.baseURI = "http://localhost";
        RestAssured.port = 8080;
        RestAssured.basePath = "/api";
    }
    
    @BeforeEach
    public void clearDatabase() {
        // Clear test data before each test
        delete("/users/all");
    }
    
    @Test
    @DisplayName("GET /users returns empty array initially")
    public void testGetUsersEmpty() {
        get("/users")
        .then()
            .statusCode(200)
            .body("size()", equalTo(0));
    }
    
    @Test
    @DisplayName("POST /users creates new user")
    public void testCreateUser() {
        String requestBody = """
            {
                "name": "John Doe",
                "email": "john@example.com"
            }
            """;
        
        given()
            .contentType("application/json")
            .body(requestBody)
        .when()
            .post("/users")
        .then()
            .statusCode(201)
            .body("name", equalTo("John Doe"))
            .body("email", equalTo("john@example.com"))
            .body("id", notNullValue());
    }

            {
                "name": "John"
            }
            """;

            .contentType("application/json")
            .body(invalidBody)

            .statusCode(400)
            .body("error", containsString("email"));
    }
    
    @Test
    @DisplayName("GET /users/:id returns user")

        .when()
            .post("/users")
        .then()
            .statusCode(201)
            .extract().response();
        
        int userId = createResponse.path("id");
        
        // Get user
        get("/users/" + userId)
        .then()
            .statusCode(200)
            .body("id", equalTo(userId))
            .body("name", equalTo("John"));
    }
    
    @Test
    @DisplayName("PUT /users/:id updates user")
    public void testUpdateUser() {
        // Create user
        int userId = given()
            .contentType("application/json")
            .body("{\"name\": \"John\", \"email\": \"john@example.com\"}")
        .when()
            .post("/users")
        .then()
            .extract().path("id");
        
        // Update user
        given()

            .body("{\"name\": \"Jane Doe\"}")
        .when()
            .put("/users/" + userId)
        .then()
            .statusCode(200)
            .body("name", equalTo("Jane Doe"));
    }
    
    @Test
    @DisplayName("DELETE /users/:id removes user")
    public void testDeleteUser() {
        // Create user
        int userId = given()

        .when()
            .post("/users")
        .then()
            .extract().path("id");
        
        // Delete user
        delete("/users/" + userId)
        .then()
            .statusCode(204);
        
        // Verify deleted
        get("/users/" + userId)
        .then()
            .statusCode(404);
    }
}
```

## Complete Testing Workflow

### 1. Test Pyramid Strategy

```
           ╱╲
          ╱  ╲         E2E Tests (Few)
         ╱────╲        - Full user workflows
        ╱      ╲       - Slow, expensive
       ╱────────╲
      ╱          ╲     Integration Tests (Some)
     ╱────────────╲    - API endpoints
    ╱              ╲   - Database interactions
   ╱────────────────╲
  ╱__________________╲ Unit Tests (Many)
                       - Business logic
                       - Fast, cheap
```

**70% Unit tests** - Test logic in isolation
**20% Integration tests** - Test API endpoints
**10% E2E tests** - Test full workflows

### 2. Typical Test Flow

```javascript
// 1. Unit tests - Fast, isolated
describe('User validation', () => {
  test('validates email format', () => {
    expect(isValidEmail('test@example.com')).toBe(true);
  });
});

// 2. Integration tests - Test API
describe('POST /users', () => {
  test('creates user', async () => {
    const response = await request(app)
      .post('/users')
      .send({ name: 'John', email: 'john@example.com' });
    expect(response.status).toBe(201);
  });
});

// 3. E2E tests - Full workflow
describe('User registration flow', () => {
  test('user can register and login', async () => {
    // 1. Register
    await register({ email: 'test@example.com', password: 'secret' });
    // 2. Verify email
    await verifyEmail(token);
    // 3. Login
    const session = await login({ email: 'test@example.com', password: 'secret' });
    expect(session.user).toBeDefined();
  });
});
```

## Key Takeaways

### Best Practices:

1. **Write tests FIRST** - TDD catches bugs early
2. **Test pyramid** - Many unit, some integration, few E2E
3. **Isolate tests** - Each test independent
4. **Use fixtures** - DRY setup code
5. **Clear names** - Describe what's being tested
6. **Fast tests** - Use mocks/stubs for external services
7. **CI/CD integration** - Run tests automatically

### Common Mistakes to Avoid:

❌ **Testing implementation** - Test behavior, not code
❌ **Slow tests** - Hitting real DB slows everything
❌ **Fragile tests** - Tests break on minor changes
❌ **No cleanup** - Tests pollute database
❌ **Too few tests** - Coverage <70%
❌ **Too many E2E tests** - Slow, expensive

### Remember:

- **Automated tests save time** - Manual testing is slower
- **Tests are documentation** - Show how API works
- **Tests enable refactoring** - Change code confidently
- **Choose tools you're comfortable with** - All tools work well
- **Testing is a skill** - Gets easier with practice

Good tests make deployments stress-free. You know immediately if something breaks, and you can fix it before users see it.
