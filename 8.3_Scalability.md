# Scalability - Building APIs That Grow

## The Problem: Why Scalability Matters

When your API launches with 100 users, a single server handles all requests easily. But what happens when you grow to 10,000 users? 100,000? 1 million?

### Real-World Scaling Challenges

#### 1. **Increased Load**
```
Day 1:  100 users  → 10 requests/sec   ✅ Server handles easily
Day 30: 1,000 users → 100 requests/sec  ⚠️  Server slowing down
Day 90: 10,000 users → 1,000 requests/sec ❌ Server overwhelmed
```

**What breaks:**
- Response times increase from 50ms to 5 seconds
- CPU maxes out at 100%
- Memory exhausted, server crashes
- Database connections saturated
- Users abandon your service

#### 2. **Single Point of Failure**
```
    All Users
       ↓
   [Single Server]  ← If this crashes, entire service is down!
       ↓
   [Database]
```

One server failure = complete outage for all users.

#### 3. **Geographic Latency**
```
User in Tokyo → Request travels 5,000 miles → Server in New York
                Takes 200ms just for network travel
```

Physics limits how fast data can travel across the globe.

---

## Understanding Scaling Strategies

### Vertical Scaling (Scaling Up)

**Definition:** Add more resources (CPU, RAM, disk) to a single server.

```
Before:                     After:
┌──────────────┐           ┌──────────────┐
│  Server      │           │  Server      │
│  4 CPUs      │    →      │  16 CPUs     │
│  8 GB RAM    │           │  64 GB RAM   │
│  100 GB Disk │           │  500 GB Disk │
└──────────────┘           └──────────────┘
```

**Advantages:**
- ✅ Simple - no code changes needed
- ✅ No distributed system complexity
- ✅ Easier to manage one server

**Disadvantages:**
- ❌ Physical limits (can't add infinite resources)
- ❌ Expensive at scale ($5k/month → $50k/month)
- ❌ Still a single point of failure
- ❌ Downtime required for upgrades
- ❌ Eventually hits a ceiling

**When to use:**
- Early stages (< 1,000 concurrent users)
- Quick temporary fix
- Database servers (harder to distribute)

**Example Implementation:**

```bash
# Start with basic server
# AWS EC2 t3.small: 2 CPUs, 2 GB RAM
# Handles ~100 concurrent users

# After a few months, vertical scale:
# Upgrade to c5.4xlarge: 16 CPUs, 32 GB RAM
# Now handles ~1,000 concurrent users

# But eventually you'll hit limits!
```

---

### Horizontal Scaling (Scaling Out)

**Definition:** Add more servers and distribute load across them.

```
Before:                     After:
    Users                       Users
      ↓                            ↓
[Single Server]         [Load Balancer]
                         ↙    ↓    ↘
                    [Server] [Server] [Server]
                        ↓       ↓       ↓
                         [Database]
```

**Advantages:**
- ✅ Nearly unlimited scalability
- ✅ No single point of failure
- ✅ Can use cheaper commodity servers
- ✅ Add/remove servers without downtime
- ✅ Better geographic distribution

**Disadvantages:**
- ❌ More complex architecture
- ❌ Requires load balancing
- ❌ Session management challenges
- ❌ Data consistency issues
- ❌ More operational overhead

**When to use:**
- Planning for significant growth
- Need high availability
- Geographic distribution required
- Cost-effective at scale

## Implementation: Horizontal Scaling Components

### 1. Load Balancing

**What is it?** Distributes incoming requests across multiple servers.

```
                 [Load Balancer]
                 Receives request:
                 GET /api/users/42
                        ↓
      ┌─────────────────┼─────────────────┐
      ↓                 ↓                 ↓
  [Server 1]        [Server 2]        [Server 3]
   30% load          35% load          35% load
      ↓                 ↓                 ↓
            [Shared Database]
```

**Load Balancing Algorithms:**

#### Round Robin
```javascript
// Simplified load balancer logic
class RoundRobinBalancer {
  constructor(servers) {
    this.servers = servers; // ['server1', 'server2', 'server3']
    this.currentIndex = 0;
  }
  
  getNextServer() {
    // Rotate through servers in order
    const server = this.servers[this.currentIndex];
    this.currentIndex = (this.currentIndex + 1) % this.servers.length;
    return server;
  }
}

// Usage
const balancer = new RoundRobinBalancer([
  'http://server1.example.com',
  'http://server2.example.com', 
  'http://server3.example.com'
]);

// Request 1 → server1
// Request 2 → server2
// Request 3 → server3
// Request 4 → server1 (wraps around)
```

**Explanation:** Each request goes to the next server in line. Simple but doesn't account for server load or health.

#### Least Connections
```javascript
// Route to server with fewest active connections
class LeastConnectionsBalancer {
  constructor(servers) {
    this.servers = servers.map(url => ({
      url,
      activeConnections: 0
    }));
  }
  
  getNextServer() {
    // Find server with minimum connections
    const server = this.servers.reduce((min, current) => 
      current.activeConnections < min.activeConnections ? current : min
    );
    
    server.activeConnections++;
    return server.url;
  }
  
  releaseConnection(serverUrl) {
    const server = this.servers.find(s => s.url === serverUrl);
    if (server) server.activeConnections--;
  }
}

// Usage
const balancer = new LeastConnectionsBalancer([
  'http://server1.example.com',
  'http://server2.example.com'
]);

async function handleRequest(req) {
  const serverUrl = balancer.getNextServer();
  try {
    const response = await fetch(`${serverUrl}${req.path}`);
    return response;
  } finally {
    balancer.releaseConnection(serverUrl);
  }
}
```

**Explanation:** Routes requests to the server handling the fewest connections. Better for handling requests of varying complexity.

#### IP Hash (Session Affinity)
```javascript
// Route same client to same server
class IPHashBalancer {
  constructor(servers) {
    this.servers = servers;
  }
  
  hashIP(ip) {
    // Simple hash function
    let hash = 0;
    for (let i = 0; i < ip.length; i++) {
      hash = ((hash << 5) - hash) + ip.charCodeAt(i);
      hash = hash & hash; // Convert to 32-bit integer
    }
    return Math.abs(hash);
  }
  
  getServer(clientIP) {
    const hash = this.hashIP(clientIP);
    const index = hash % this.servers.length;
    return this.servers[index];
  }
}

// Usage
const balancer = new IPHashBalancer([
  'http://server1.example.com',
  'http://server2.example.com',
  'http://server3.example.com'
]);

// User with IP 192.168.1.100 always goes to same server
const server = balancer.getServer('192.168.1.100');
```

**Explanation:** Same client always hits the same server. Useful for maintaining server-side sessions, but reduces flexibility.

### 2. Stateless Design

**The Golden Rule:** Your API servers should store NO session data locally.

#### ❌ Bad: Stateful Server
```javascript
// DON'T DO THIS - Storing session in server memory
const sessions = {}; // Stored on THIS server only

app.post('/api/login', (req, res) => {
  const { username, password } = req.body;
  
  if (authenticate(username, password)) {
    const sessionId = generateSessionId();
    
    // ❌ Problem: Stored only on this server!
    sessions[sessionId] = {
      username,
      loginTime: Date.now()
    };
    
    res.json({ sessionId });
  }
});

app.get('/api/profile', (req, res) => {
  const { sessionId } = req.headers;
  
  // ❌ Problem: What if request goes to different server?
  const session = sessions[sessionId]; // Won't exist on other servers!
  
  if (!session) {
    return res.status(401).json({ error: 'Not authenticated' });
  }
  
  res.json({ username: session.username });
});
```

**What breaks in scaled environment:**
```
User logs in → Request goes to Server 1
              Server 1 stores session in memory ✅

User requests profile → Load balancer sends to Server 2
                       Server 2 has no session data ❌
                       User gets 401 error!
```

#### ✅ Good: Stateless Server
```javascript
// Store session in shared Redis cache
const redis = require('redis');
const client = redis.createClient({
  host: 'redis-cluster.example.com'
});

app.post('/api/login', async (req, res) => {
  const { username, password } = req.body;
  
  if (authenticate(username, password)) {
    const sessionId = generateSessionId();
    
    // ✅ Store in shared Redis - accessible by all servers
    await client.set(
      `session:${sessionId}`,
      JSON.stringify({
        username,
        loginTime: Date.now()
      }),
      'EX', 3600 // Expires in 1 hour
    );
    
    res.json({ sessionId });
  }
});

app.get('/api/profile', async (req, res) => {
  const { sessionId } = req.headers;
  
  // ✅ Any server can retrieve session from Redis
  const sessionData = await client.get(`session:${sessionId}`);
  
  if (!sessionData) {
    return res.status(401).json({ error: 'Not authenticated' });
  }
  
  const session = JSON.parse(sessionData);
  res.json({ username: session.username });
});
```

**How this scales:**
```
        [Server 1]    [Server 2]    [Server 3]
             ↓             ↓             ↓
          [Shared Redis Cache]
          Contains all sessions
          Accessible by any server
```

**Even better: JWT (Completely Stateless)**
```javascript
const jwt = require('jsonwebtoken');
const SECRET = process.env.JWT_SECRET;

app.post('/api/login', (req, res) => {
  const { username, password } = req.body;
  
  if (authenticate(username, password)) {
    // ✅ Create self-contained token
    const token = jwt.sign(
      { 
        username,
        role: 'user',
        exp: Math.floor(Date.now() / 1000) + (60 * 60) // 1 hour
      },
      SECRET
    );
    
    // No server-side storage needed!
    res.json({ token });
  }
});

app.get('/api/profile', (req, res) => {
  const token = req.headers.authorization?.replace('Bearer ', '');
  
  try {
    // ✅ Token contains all needed data - no database/cache lookup!
    const decoded = jwt.verify(token, SECRET);
    
    res.json({ 
      username: decoded.username,
      role: decoded.role 
    });
  } catch (error) {
    res.status(401).json({ error: 'Invalid token' });
  }
});
```

**Explanation:** JWT tokens contain all authentication data. No server-side state needed. Any server can validate the token independently.

### 3. Database Replication

**The Problem:** Database is often the bottleneck.

```
    [Server 1]    [Server 2]    [Server 3]
         ↓             ↓             ↓
    [Single Database]  ← Bottleneck! Can't handle load
```

**Solution: Read Replicas**

```javascript
// Primary database for writes, replicas for reads
const mysql = require('mysql2/promise');

// Primary database - handles all writes
const primaryDb = mysql.createPool({
  host: 'primary-db.example.com',
  user: 'api',
  database: 'users',
  connectionLimit: 10
});

// Read replicas - handle read queries
const readReplicas = [
  mysql.createPool({
    host: 'replica1-db.example.com',
    user: 'api',
    database: 'users',
    connectionLimit: 20
  }),
  mysql.createPool({
    host: 'replica2-db.example.com',
    user: 'api',
    database: 'users',
    connectionLimit: 20
  }),
  mysql.createPool({
    host: 'replica3-db.example.com',
    user: 'api',
    database: 'users',
    connectionLimit: 20
  })
];

// Round-robin through read replicas
let replicaIndex = 0;
function getReadReplica() {
  const replica = readReplicas[replicaIndex];
  replicaIndex = (replicaIndex + 1) % readReplicas.length;
  return replica;
}

// GET request - use read replica
app.get('/api/users/:id', async (req, res) => {
  const db = getReadReplica(); // ✅ Use replica for reads
  
  const [users] = await db.query(
    'SELECT * FROM users WHERE id = ?',
    [req.params.id]
  );
  
  res.json(users[0]);
});

// POST request - use primary
app.post('/api/users', async (req, res) => {
  const db = primaryDb; // ✅ Use primary for writes
  
  const [result] = await db.query(
    'INSERT INTO users (name, email) VALUES (?, ?)',
    [req.body.name, req.body.email]
  );
  
  res.status(201).json({ id: result.insertId });
});

// PUT request - use primary
app.put('/api/users/:id', async (req, res) => {
  const db = primaryDb; // ✅ Use primary for writes
  
  await db.query(
    'UPDATE users SET name = ?, email = ? WHERE id = ?',
    [req.body.name, req.body.email, req.params.id]
  );
  
  res.json({ success: true });
});
```

**How it works:**
```
Writes (20% of traffic):
  API → Primary Database → Replicates to replicas

Reads (80% of traffic):
  API → Replica 1 (33%)
      → Replica 2 (33%)
      → Replica 3 (33%)
```

**Explanation:** Most applications read data 4-5x more than they write. By routing reads to replicas, you can handle much higher load.

**Important: Replication Lag**
```javascript
// Be aware of replication lag
app.post('/api/users', async (req, res) => {
  // Write to primary
  const [result] = await primaryDb.query(
    'INSERT INTO users (name, email) VALUES (?, ?)',
    [req.body.name, req.body.email]
  );
  
  const userId = result.insertId;
  
  // ❌ DON'T immediately read from replica
  // Replication might not have completed yet!
  
  // ✅ DO read from primary for just-created data
  const [users] = await primaryDb.query(
    'SELECT * FROM users WHERE id = ?',
    [userId]
  );
  
  res.status(201).json(users[0]);
});
```

### 4. Caching Layer

**Reduce database load by caching frequently accessed data.**

```javascript
const Redis = require('ioredis');
const redis = new Redis({
  host: 'redis-cluster.example.com',
  keyPrefix: 'api:'
});

// Cache-aside pattern
app.get('/api/users/:id', async (req, res) => {
  const userId = req.params.id;
  const cacheKey = `user:${userId}`;
  
  // 1. Try cache first
  const cached = await redis.get(cacheKey);
  if (cached) {
    console.log('Cache HIT');
    return res.json(JSON.parse(cached));
  }
  
  console.log('Cache MISS');
  
  // 2. Cache miss - query database
  const [users] = await readReplica.query(
    'SELECT * FROM users WHERE id = ?',
    [userId]
  );
  
  const user = users[0];
  
  if (user) {
    // 3. Store in cache for 5 minutes
    await redis.setex(
      cacheKey,
      300, // TTL: 5 minutes
      JSON.stringify(user)
    );
  }
  
  res.json(user);
});

// Invalidate cache on updates
app.put('/api/users/:id', async (req, res) => {
  const userId = req.params.id;
  
  // 1. Update database
  await primaryDb.query(
    'UPDATE users SET name = ?, email = ? WHERE id = ?',
    [req.body.name, req.body.email, userId]
  );
  
  // 2. Invalidate cache
  await redis.del(`user:${userId}`);
  
  res.json({ success: true });
});
```

**Cache Performance Impact:**
```
Without cache:
  1000 requests/sec × 50ms database query = High DB load
  Database struggles

With cache (90% hit rate):
  100 requests → Database (10%)
  900 requests → Cache (90%) ← Served in <1ms!
  Database easily handles remaining 10%
```

**Explanation:** Caching reduces database load by 10-100x depending on your cache hit rate. Essential for scaling.

### 5. Microservices Architecture

**Break monolith into independently scalable services.**

#### Before: Monolithic API
```
           [Single API Server]
               ↓
    ┌──────────┼──────────┐
    ↓          ↓          ↓
  Users    Products    Orders
  Service  Service    Service
    ↓          ↓          ↓
         [Database]
```

**Problem:** If Orders service gets 10x traffic, you must scale the ENTIRE server (including Users and Products).

#### After: Microservices
```javascript
// User Service - Scales independently
// users-service.js (Port 3001)
const express = require('express');
const app = express();

app.get('/api/users/:id', async (req, res) => {
  const user = await UserModel.findById(req.params.id);
  res.json(user);
});

app.listen(3001);

// Product Service - Scales independently  
// products-service.js (Port 3002)
const express = require('express');
const app = express();

app.get('/api/products/:id', async (req, res) => {
  const product = await ProductModel.findById(req.params.id);
  res.json(product);
});

app.listen(3002);

// Order Service - Scales independently
// orders-service.js (Port 3003)
const express = require('express');
const app = express();

app.post('/api/orders', async (req, res) => {
  // Call other services via HTTP
  const user = await fetch(`http://users-service:3001/api/users/${req.body.userId}`);
  const product = await fetch(`http://products-service:3002/api/products/${req.body.productId}`);
  
  const order = await OrderModel.create({
    userId: req.body.userId,
    productId: req.body.productId,
    total: (await product.json()).price
  });
  
  res.status(201).json(order);
});

app.listen(3003);
```

**Deployment:**
```
User Service:     [S1] [S2]           ← 2 instances
Product Service:  [S1] [S2] [S3] [S4] ← 4 instances (high demand)
Order Service:    [S1] [S2] [S3]      ← 3 instances (Black Friday surge)
```

**Explanation:** Each service scales independently based on its own traffic patterns. Orders service can scale to 10 instances during Black Friday without affecting Users service.

### 6. Content Delivery Network (CDN)

**Distribute static assets globally for faster access.**

```javascript
// Serve user avatars from CDN
app.get('/api/users/:id', async (req, res) => {
  const user = await UserModel.findById(req.params.id);
  
  // ❌ Don't serve from your server
  // avatarUrl: `https://api.example.com/avatars/${user.avatarFile}`
  
  // ✅ Use CDN
  res.json({
    id: user.id,
    name: user.name,
    avatarUrl: `https://cdn.example.com/avatars/${user.avatarFile}`
  });
});
```

**How CDN works:**
```
User in Tokyo → CDN Edge Server (Tokyo) → Serves cached file in 10ms
                      ↑
                First time: Fetches from origin server (200ms)
                Caches for future requests
```

**Explanation:** CDN caches files at edge locations worldwide. User downloads from nearest location, reducing latency from 200ms to 10ms.

## Complete Scaling Example

**Scaling from 100 to 100,000 concurrent users:**

```javascript
// Stage 1: Single Server (0-1K users)
// server.js - Basic Express app
const express = require('express');
const app = express();

app.get('/api/users/:id', async (req, res) => {
  const user = await db.query('SELECT * FROM users WHERE id = ?', [req.params.id]);
  res.json(user);
});

app.listen(3000);
```

```javascript
// Stage 2: Add Database Replicas (1K-10K users)
// Add read replicas to distribute database load
const readReplica = getReadReplica();

app.get('/api/users/:id', async (req, res) => {
  const user = await readReplica.query('SELECT * FROM users WHERE id = ?', [req.params.id]);
  res.json(user);
});
```

```javascript
// Stage 3: Add Caching (10K-50K users)
// Reduce database queries with Redis
app.get('/api/users/:id', async (req, res) => {
  const cached = await redis.get(`user:${req.params.id}`);
  if (cached) return res.json(JSON.parse(cached));
  
  const user = await readReplica.query('SELECT * FROM users WHERE id = ?', [req.params.id]);
  await redis.setex(`user:${req.params.id}`, 300, JSON.stringify(user));
  res.json(user);
});
```

```javascript
// Stage 4: Horizontal Scaling + Load Balancer (50K-100K users)
// Deploy multiple servers behind load balancer

// nginx.conf - Load balancer configuration
upstream api_servers {
  least_conn;
  server api1.example.com:3000;
  server api2.example.com:3000;
  server api3.example.com:3000;
  server api4.example.com:3000;
  server api5.example.com:3000;
}

server {
  listen 80;
  location /api/ {
    proxy_pass http://api_servers;
  }
}
```

**Final Architecture:**
```
                   [CDN]
                     ↓
              [Load Balancer]
         ↙      ↓      ↓      ↘
    [API 1] [API 2] [API 3] [API 4]
         ↓       ↓       ↓       ↓
    [Redis Cache Layer]
         ↓       ↓       ↓       ↓
   [Primary DB] → [Replica 1] [Replica 2]
```

## Key Takeaways

### When to Use Each Strategy:

| Users | Strategy | Why |
|-------|----------|-----|
| 0-1K | Single server | Simple, cost-effective |
| 1K-5K | Vertical scaling + caching | Quick wins before complexity |
| 5K-20K | Database replicas | Database often bottleneck |
| 20K-100K | Horizontal scaling + load balancer | Need redundancy |
| 100K+ | Microservices + CDN | Full distributed architecture |

### Scaling Principles:

1. **Measure first** - Use monitoring to identify bottlenecks
2. **Scale incrementally** - Don't over-engineer early
3. **Design stateless** - Makes horizontal scaling possible
4. **Cache aggressively** - Reduces database load 10-100x
5. **Distribute data** - Read replicas, sharding, CDN
6. **Plan for failure** - Redundancy at every layer

### Remember:
- Vertical scaling is faster to implement but hits limits
- Horizontal scaling is harder but scales infinitely
- Most apps need both at different stages
- Premature optimization wastes time - scale when needed
- Monitor everything to know when to scale
