# 10.1 Alternative API Styles

## The Problem

You've learned REST thoroughly. But REST isn't always the best choice. Different problems call for different solutions.

**Common scenarios where REST struggles:**

1. **Mobile apps with limited bandwidth**: Fetching data requires multiple REST requests, wasting bandwidth and battery
2. **Real-time applications**: Polling REST endpoints is inefficient for live updates
3. **Microservices with strict contracts**: REST's flexibility makes it hard to enforce type safety
4. **High-performance systems**: JSON over HTTP has overhead; binary protocols are faster
5. **Over-fetching/under-fetching**: REST returns fixed structures; clients often need more or less data

**The landscape:**
- **REST**: Flexible, web-friendly, most common
- **GraphQL**: Client-driven queries, solve over/under-fetching
- **gRPC**: High-performance binary protocol, strong typing
- **SOAP**: Enterprise standard, strict contracts, legacy systems
- **WebSocket**: Real-time bidirectional communication

This guide explains each alternative, compares it to REST, and helps you choose the right tool for your needs.

---

## Understanding the Architectural Styles

### Quick Comparison Table

| Style | Protocol | Data Format | Use Case | Complexity | Performance |
|-------|----------|-------------|----------|------------|-------------|
| **REST** | HTTP/1.1 | JSON/XML | General-purpose APIs | Low | Good |
| **GraphQL** | HTTP/1.1 | JSON | Flexible data fetching | Medium | Good |
| **gRPC** | HTTP/2 | Protocol Buffers | Microservices, high-perf | High | Excellent |
| **SOAP** | HTTP/SMTP | XML | Enterprise, legacy | High | Fair |
| **WebSocket** | WS/WSS | Any | Real-time, bidirectional | Medium | Excellent |

Now let's explore each in detail.

## Alternative 1: GraphQL

### What is GraphQL?

**GraphQL** is a query language for APIs where clients specify exactly what data they need. Instead of multiple REST endpoints, you have one endpoint that handles all queries.

**Core principle:** Client-driven data fetching.

### The REST Problem GraphQL Solves

**REST example - The over-fetching problem:**

```javascript
// Scenario: Show user's name and their latest 3 posts

// REST approach - Multiple requests
// Request 1: Get user
GET /api/users/123
Response: {
  id: 123,
  name: "Alice",
  email: "alice@example.com",
  bio: "Long bio text...",
  avatar: "https://...",
  settings: { /* lots of unused data */ },
  created_at: "2020-01-01"
  // 50 more fields you don't need...
}

// Request 2: Get user's posts
GET /api/users/123/posts?limit=3
Response: [
  {
    id: 1,
    title: "Post 1",
    body: "Full post content...",  // Don't need this
    author: { /* duplicate user data */ },
    comments: [ /* don't need comments */ ],
    likes_count: 42,
    created_at: "2024-01-15"
    // More fields you don't need...
  },
  // ... 2 more posts
]

// Problems:
// ❌ 2 HTTP requests (latency, especially on mobile)
// ❌ Over-fetching: Got 50 user fields, needed only 'name'
// ❌ Over-fetching: Got full post content, only needed title
// ❌ Can't customize: Fixed structure from server
```

**GraphQL solution - One request, exactly what you need:**

```javascript
// GraphQL approach - Single request
POST /graphql
{
  query: `
    query {
      user(id: 123) {
        name
        posts(limit: 3) {
          title
          created_at
        }
      }
    }
  `
}

Response: {
  "data": {
    "user": {
      "name": "Alice",
      "posts": [
        { "title": "Post 1", "created_at": "2024-01-15" },
        { "title": "Post 2", "created_at": "2024-01-14" },
        { "title": "Post 3", "created_at": "2024-01-13" }
      ]
    }
  }
}

// Benefits:
// ✅ 1 HTTP request (faster, better for mobile)
// ✅ Exactly the fields needed (less bandwidth)
// ✅ Client controls what data to fetch
// ✅ Nested data in single request
```

### How GraphQL Works

**REST vs GraphQL architecture:**

```
REST:
┌────────┐              ┌────────┐
│ Client │─ GET /users ─>│ Server │ (returns ALL user fields)
└────────┘              └────────┘
┌────────┐              ┌────────┐
│ Client │─ GET /posts ─>│ Server │ (returns ALL post fields)
└────────┘              └────────┘
Multiple endpoints, fixed responses

GraphQL:
┌────────┐              ┌────────┐
│ Client │─ POST /graphql ─>│ Server │
│  query:{│              │Resolvers│ (returns ONLY requested fields)
│  user {name}          │        │
│  posts {title}        │        │
│  }                    │        │
└────────┘              └────────┘
Single endpoint, client defines response
```

### Setting Up GraphQL Server (Node.js)

```javascript
// Step 1: Install dependencies
// npm install apollo-server graphql

const { ApolloServer, gql } = require('apollo-server');

// Step 2: Define schema (what data is available)
const typeDefs = gql`
  # User type definition
  type User {
    id: ID!           # ! means required
    name: String!
    email: String!
    bio: String
    posts: [Post!]!   # Array of posts
  }

  # Post type definition
  type Post {
    id: ID!
    title: String!
    body: String!
    author: User!     # Nested relationship
    created_at: String!
  }

  # Queries (read operations)
  type Query {
    # Get single user by ID
    user(id: ID!): User
    
    # Get all users
    users: [User!]!
    
    # Get single post
    post(id: ID!): Post
    
    # Get all posts
    posts(limit: Int): [Post!]!
    
    # Search posts
    searchPosts(keyword: String!): [Post!]!
  }

  # Mutations (write operations)
  type Mutation {
    # Create new user
    createUser(name: String!, email: String!, bio: String): User!
    
    # Create new post
    createPost(title: String!, body: String!, authorId: ID!): Post!
    
    # Update post
    updatePost(id: ID!, title: String, body: String): Post!
    
    # Delete post
    deletePost(id: ID!): Boolean!
  }
`;

// Step 3: Define resolvers (how to fetch the data)
const resolvers = {
  Query: {
    // Resolver for user query
    user: async (parent, { id }, context) => {
      // Fetch from database
      return await db.users.findById(id);
    },

    users: async () => {
      return await db.users.findAll();
    },

    posts: async (parent, { limit }) => {
      return await db.posts.findAll({ limit });
    },

    searchPosts: async (parent, { keyword }) => {
      return await db.posts.search(keyword);
    }
  },

  Mutation: {
    createUser: async (parent, { name, email, bio }) => {
      return await db.users.create({ name, email, bio });
    },

    createPost: async (parent, { title, body, authorId }) => {
      return await db.posts.create({ title, body, authorId });
    },

    updatePost: async (parent, { id, title, body }) => {
      return await db.posts.update(id, { title, body });
    },

    deletePost: async (parent, { id }) => {
      await db.posts.delete(id);
      return true;
    }
  },

  // Nested resolvers (for relationships)
  User: {
    // When a user's posts are requested
    posts: async (user) => {
      return await db.posts.findByUserId(user.id);
    }
  },

  Post: {
    // When a post's author is requested
    author: async (post) => {
      return await db.users.findById(post.authorId);
    }
  }
};

// Step 4: Create and start server
const server = new ApolloServer({
  typeDefs,
  resolvers,
  context: ({ req }) => {
    // Add authentication, user info, etc.
    return {
      user: req.user,
      db: db
    };
  }
});

server.listen().then(({ url }) => {
  console.log(`GraphQL server ready at ${url}`);
});
```

**How it works:**
1. **Schema** defines what data is available (types, queries, mutations)
2. **Resolvers** define how to fetch that data
3. Client sends **query** specifying what fields it needs
4. GraphQL executes only the resolvers needed
5. Returns data in exact shape requested

### GraphQL Client Usage

```javascript
// Client-side - Querying data
const query = `
  query GetUserDashboard($userId: ID!) {
    user(id: $userId) {
      name
      email

        id
        title
        created_at
      }
    }
  }
`;

const variables = { userId: "123" };

const response = await fetch('/graphql', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ query, variables })
});

const data = await response.json();
// { data: { user: { name: "Alice", email: "...", posts: [...] } } }
```

```javascript
// Mutation example - Creating data
const mutation = `
  mutation CreatePost($title: String!, $body: String!, $authorId: ID!) {
    createPost(title: $title, body: $body, authorId: $authorId) {
      id
      title
      created_at
      author {
        name
      }
    }
  }
`;

const variables = {
  title: "My New Post",
  body: "Post content here...",
  authorId: "123"
};

const response = await fetch('/graphql', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ query: mutation, variables })
});
```

### GraphQL Advanced: Fragments, Aliases, Directives

```javascript
// Fragments - Reusable field sets
const query = `
  fragment UserFields on User {
    id
    name
    email
  }

  query {
    user1: user(id: "123") {
      ...UserFields
    }
    user2: user(id: "456") {
      ...UserFields
    }
  }
`;

// Aliases - Query same field multiple times
const query = `
  query {
    recentPosts: posts(limit: 5, sortBy: "created_at") {
      title
    }
    popularPosts: posts(limit: 5, sortBy: "likes") {
      title
    }
  }
`;

// Directives - Conditional fields
const query = `
  query GetUser($id: ID!, $withPosts: Boolean!) {
    user(id: $id) {
      name
      email
      posts @include(if: $withPosts) {
        title
      }
    }
  }
`;
// If withPosts=false, posts won't be fetched
```

### GraphQL vs REST: Side-by-Side

**Scenario:** Build a mobile app showing user profile with recent posts and comments

**REST approach:**
```javascript
// 3 requests needed
GET /api/users/123           // Get user info
GET /api/users/123/posts?limit=10  // Get user's posts
GET /api/posts/456/comments  // Get comments for one post

// Total: 3 HTTP roundtrips
// Over-fetching: Got all user fields, all post fields
// Under-fetching: Need separate request for each post's comments
```

**GraphQL approach:**
```javascript
// 1 request
query {
  user(id: "123") {
    name
    avatar
    posts(limit: 10) {
      title
      likes
      comments {
        text
        author {
          name
        }
      }
    }
  }
}

// Total: 1 HTTP roundtrip
// Exactly the fields needed
// Nested comments fetched in same request
```

### When to Choose GraphQL

**✅ Choose GraphQL when:**
- Building mobile apps (reduce network requests, save bandwidth)
- Frontend needs vary (different pages need different data)
- Nested/related data is common (users → posts → comments)
- You have many clients with different needs (web, mobile, desktop)
- Rapid frontend iteration (no backend changes for new fields)
- Over-fetching/under-fetching is a problem

**❌ Avoid GraphQL when:**
- Simple CRUD operations (REST is simpler)
- Clients need same data (no customization needed)
- File uploads/downloads (GraphQL has limited support)
- Strict caching requirements (HTTP caching harder with POST)
- Team lacks GraphQL expertise (learning curve)
- Small project (overhead not worth it)

### GraphQL Pros and Cons

**Pros:**
- ✅ Eliminates over-fetching (client requests only needed fields)
- ✅ Single request for nested data (better for mobile)
- ✅ Strong typing (schema acts as contract)
- ✅ Self-documenting (introspection shows all available data)
- ✅ Frontend flexibility (no backend changes for new queries)
- ✅ Great developer experience (GraphQL Playground, autocomplete)

**Cons:**
- ❌ Complexity (more complex than REST)
- ❌ HTTP caching harder (everything is POST /graphql)
- ❌ Query complexity (clients can write expensive queries)
- ❌ N+1 query problem (need DataLoader to avoid)
- ❌ File uploads require special handling
- ❌ Learning curve for team

## Alternative 2: gRPC

### What is gRPC?

**gRPC** (Google Remote Procedure Call) is a high-performance framework using HTTP/2 and Protocol Buffers. Instead of resources (REST), you define services with methods.

**Core principle:** Call remote functions as if they were local.

### The REST Problem gRPC Solves

**REST limitations:**
```javascript
// REST is text-based (JSON), which is:
// ❌ Slow to parse (CPU intensive)
// ❌ Large payload size (verbose)
// ❌ No type safety (runtime errors)
// ❌ HTTP/1.1 limitations (one request per connection)

// REST request/response
POST /api/users HTTP/1.1
Content-Type: application/json

{"name":"Alice","email":"alice@example.com","age":30}

HTTP/1.1 201 Created
{"id":"123","name":"Alice","email":"alice@example.com","age":30,"created_at":"2024-01-15"}

// Issues for high-traffic microservices:
// - JSON parsing overhead at scale
// - Larger payloads (more bandwidth)
// - No contract enforcement (typos cause runtime errors)
```

**gRPC solution:**
```protobuf
// Binary protocol with strong typing
// ✅ Fast binary serialization (10x faster than JSON)
// ✅ Compact (50% smaller payloads)
// ✅ Type-safe (compile-time errors)
// ✅ HTTP/2 (multiplexing, streaming)

// Define service contract
service UserService {
  rpc CreateUser(CreateUserRequest) returns (User);
  rpc GetUser(GetUserRequest) returns (User);
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
  int32 age = 3;
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
  int32 age = 4;
  string created_at = 5;
}
```

### How gRPC Works

**Architecture:**
```
┌────────────┐                    ┌────────────┐
│   Client   │                    │   Server   │
│            │                    │            │
│ Call:      │                    │ Implements:│
│ GetUser()  │──binary message──>│ GetUser()  │
│            │<──binary response──│            │
└────────────┘    HTTP/2          └────────────┘
     ↑                                  ↑
     └── Generated from .proto file ────┘
```

**Key differences from REST:**
1. **Binary, not text**: Protocol Buffers instead of JSON
2. **RPC, not REST**: Call methods, not HTTP verbs on resources
3. **HTTP/2**: Multiplexing, streaming, server push
4. **Code generation**: Client and server code auto-generated from .proto file

### Setting Up gRPC (Node.js)

```protobuf
// Step 1: Define service in users.proto file
syntax = "proto3";

package users;

// Service definition
service UserService {
  // Unary RPC (single request, single response)
  rpc GetUser(GetUserRequest) returns (User);
  
  // Server streaming (single request, stream of responses)
  rpc ListUsers(ListUsersRequest) returns (stream User);
  
  // Client streaming (stream of requests, single response)
  rpc CreateUsers(stream CreateUserRequest) returns (CreateUsersResponse);
  
  // Bidirectional streaming
  rpc Chat(stream ChatMessage) returns (stream ChatMessage);
}

// Message types
message GetUserRequest {
  string user_id = 1;
}

message ListUsersRequest {
  int32 page_size = 1;
  string page_token = 2;
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
  int32 age = 3;
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
  int32 age = 4;
  string created_at = 5;
}

message CreateUsersResponse {
  int32 count = 1;
}

message ChatMessage {
  string user_id = 1;
  string message = 2;
  string timestamp = 3;
}
```

```javascript
// Step 2: Implement gRPC server
const grpc = require('@grpc/grpc-js');
const protoLoader = require('@grpc/proto-loader');

// Load proto file
const packageDefinition = protoLoader.loadSync('users.proto', {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true
});

const usersProto = grpc.loadPackageDefinition(packageDefinition).users;

// Implement service methods
const userService = {
  // Unary RPC
  GetUser: async (call, callback) => {
    try {
      const userId = call.request.user_id;
      const user = await db.users.findById(userId);
      
      if (!user) {
        callback({
          code: grpc.status.NOT_FOUND,
          details: 'User not found'
        });
        return;
      }
      
      callback(null, {
        id: user.id,
        name: user.name,
        email: user.email,
        age: user.age,
        created_at: user.createdAt.toISOString()
      });
    } catch (error) {
      callback({
        code: grpc.status.INTERNAL,
        details: error.message
      });
    }
  },

  // Server streaming RPC
  ListUsers: async (call) => {
    const pageSize = call.request.page_size || 10;
    
    // Stream users one by one
    const users = await db.users.findAll({ limit: pageSize });
    
    for (const user of users) {
      call.write({
        id: user.id,
        name: user.name,
        email: user.email,
        age: user.age,
        created_at: user.createdAt.toISOString()
      });
    }
    
    call.end(); // Done streaming
  },

  // Client streaming RPC
  CreateUsers: (call, callback) => {
    const users = [];
    
    // Receive stream of users
    call.on('data', (createUserRequest) => {
      users.push({
        name: createUserRequest.name,
        email: createUserRequest.email,
        age: createUserRequest.age
      });
    });
    
    // When client finishes streaming
    call.on('end', async () => {
      await db.users.bulkCreate(users);
      callback(null, { count: users.length });
    });
  },

  // Bidirectional streaming RPC
  Chat: (call) => {
    call.on('data', (message) => {
      console.log('Received:', message);
      
      // Echo back to all clients (simplified)
      call.write({
        user_id: message.user_id,
        message: message.message,
        timestamp: new Date().toISOString()
      });
    });
    
    call.on('end', () => {
      call.end();
    });
  }
};

// Create and start server
const server = new grpc.Server();
server.addService(usersProto.UserService.service, userService);

server.bindAsync(
  '0.0.0.0:50051',
  grpc.ServerCredentials.createInsecure(),
  (error, port) => {
    if (error) {
      console.error(error);
      return;
    }
    console.log(`gRPC server running on port ${port}`);
    server.start();
  }
);
```

```javascript
// Step 3: Create gRPC client
const grpc = require('@grpc/grpc-js');
const protoLoader = require('@grpc/proto-loader');

const packageDefinition = protoLoader.loadSync('users.proto');
const usersProto = grpc.loadPackageDefinition(packageDefinition).users;

// Create client
const client = new usersProto.UserService(
  'localhost:50051',
  grpc.credentials.createInsecure()
);

// Unary call
client.GetUser({ user_id: '123' }, (error, user) => {
  if (error) {
    console.error('Error:', error);
    return;
  }
  console.log('User:', user);
});

// Server streaming
const call = client.ListUsers({ page_size: 10 });

call.on('data', (user) => {
  console.log('Received user:', user);
});

call.on('end', () => {
  console.log('Stream ended');
});

call.on('error', (error) => {
  console.error('Stream error:', error);
});

// Client streaming
const createUsersCall = client.CreateUsers((error, response) => {
  if (error) {
    console.error('Error:', error);
    return;
  }
  console.log(`Created ${response.count} users`);
});

// Send stream of users
createUsersCall.write({ name: 'Alice', email: 'alice@example.com', age: 30 });
createUsersCall.write({ name: 'Bob', email: 'bob@example.com', age: 25 });
createUsersCall.write({ name: 'Carol', email: 'carol@example.com', age: 35 });
createUsersCall.end();
```

### gRPC vs REST Performance

```javascript
// Benchmark comparison (approximate):

// REST (JSON over HTTP/1.1)
// Payload size: 250 bytes
// Parse time: 0.5ms
// Throughput: 10,000 requests/sec

// gRPC (Protocol Buffers over HTTP/2)
// Payload size: 100 bytes (60% smaller)
// Parse time: 0.05ms (10x faster)
// Throughput: 50,000 requests/sec (5x faster)

// Why gRPC is faster:
// 1. Binary encoding (no JSON parsing)
// 2. Smaller payloads (less network transfer)
// 3. HTTP/2 multiplexing (multiple streams over one connection)
// 4. Header compression (HPACK)
```

### When to Choose gRPC

**✅ Choose gRPC when:**
- Microservices communication (internal services)
- High-performance requirements (thousands of requests/sec)
- Polyglot environment (auto-generate clients in many languages)
- Streaming needed (real-time data, large file transfer)
- Strong typing desired (catch errors at compile-time)
- Low latency critical (trading systems, gaming, IoT)

**❌ Avoid gRPC when:**
- Browser clients (limited browser support)
- External/public APIs (REST is more accessible)
- Simple CRUD (gRPC is overkill)
- Human-readable debugging (binary is not readable)
- Team lacks Protocol Buffers expertise
- Need HTTP caching (gRPC doesn't use standard HTTP caching)

### gRPC Pros and Cons

**Pros:**
- ✅ High performance (binary, HTTP/2, faster than REST)
- ✅ Compact payloads (50% smaller than JSON)
- ✅ Strongly typed (Protocol Buffers schema)
- ✅ Code generation (clients/servers auto-generated)
- ✅ Streaming support (bidirectional, client, server)
- ✅ Polyglot (works with 10+ languages)

**Cons:**
- ❌ Limited browser support (need gRPC-Web proxy)
- ❌ Not human-readable (binary protocol)
- ❌ Complexity (steeper learning curve)
- ❌ Ecosystem smaller than REST
- ❌ Debugging harder (binary data)
- ❌ Not suitable for public APIs

## Alternative 3: SOAP

### What is SOAP?

**SOAP** (Simple Object Access Protocol) is an XML-based protocol with strict standards. It's the predecessor to REST, still used in enterprise and legacy systems.

**Core principle:** Strict contract, enterprise features (security, transactions).

### SOAP Overview

**SOAP is strict and formal:**
```xml
<!-- SOAP request -->
POST /api HTTP/1.1
Content-Type: text/xml

<?xml version="1.0"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Header>
    <auth:Authentication xmlns:auth="http://example.com/auth">
      <auth:Username>admin</auth:Username>
      <auth:Password>secret</auth:Password>
    </auth:Authentication>
  </soap:Header>
  <soap:Body>
    <m:GetUser xmlns:m="http://example.com/users">
      <m:UserId>123</m:UserId>
    </m:GetUser>
  </soap:Body>
</soap:Envelope>

<!-- SOAP response -->
<?xml version="1.0"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <m:GetUserResponse xmlns:m="http://example.com/users">
      <m:User>
        <m:Id>123</m:Id>
        <m:Name>Alice</m:Name>
        <m:Email>alice@example.com</m:Email>
      </m:User>
    </m:GetUserResponse>
  </soap:Body>
</soap:Envelope>
```

### SOAP vs REST: Quick Comparison

| Aspect | SOAP | REST |
|--------|------|------|
| **Protocol** | Strict protocol | Architectural style |
| **Format** | XML only | JSON, XML, others |
| **Payload** | Verbose (XML) | Lightweight (JSON) |
| **Contract** | WSDL (strict) | Optional (OpenAPI) |
| **Security** | WS-Security (built-in) | OAuth, JWT (add-on) |
| **Transactions** | WS-AtomicTransaction | Not built-in |
| **Statefulness** | Can be stateful | Stateless |
| **Complexity** | High | Low |
| **Use Case** | Enterprise, legacy | Modern web/mobile |

### SOAP Example (Node.js)

```javascript
// SOAP server using 'soap' package
const soap = require('soap');
const http = require('http');

// Define service
const service = {
  UserService: {
    UserServicePort: {

          User: {
            Id: user.id,

      CreateUser: async (args) => {
        const user = await db.users.create({
          name: args.Name,
          email: args.Email
        });
        
        return {
          User: {
            Id: user.id,
            Name: user.name,
            Email: user.email
          }
        };
      }
    }
  }
};

// WSDL (Web Services Description Language) - defines contract
const wsdl = `
<?xml version="1.0"?>
<definitions name="UserService"
  targetNamespace="http://example.com/users"
  xmlns="http://schemas.xmlsoap.org/wsdl/"
  xmlns:soap="http://schemas.xmlsoap.org/wsdl/soap/"
  xmlns:tns="http://example.com/users"
  xmlns:xsd="http://www.w3.org/2001/XMLSchema">

  <message name="GetUserRequest">
    <part name="UserId" type="xsd:string"/>
  </message>

  <message name="GetUserResponse">
    <part name="User" type="tns:User"/>
  </message>

  <portType name="UserServicePort">
    <operation name="GetUser">
      <input message="tns:GetUserRequest"/>
      <output message="tns:GetUserResponse"/>
    </operation>
  </portType>

  <binding name="UserServiceBinding" type="tns:UserServicePort">
    <soap:binding transport="http://schemas.xmlsoap.org/soap/http"/>
    <operation name="GetUser">
      <soap:operation soapAction="GetUser"/>
      <input><soap:body use="literal"/></input>
      <output><soap:body use="literal"/></output>
    </operation>
  </binding>

  <service name="UserService">
    <port name="UserServicePort" binding="tns:UserServiceBinding">
      <soap:address location="http://localhost:8000/soap"/>
    </port>
  </service>
</definitions>
`;

const server = http.createServer((req, res) => {
  res.end("SOAP service running");
});

server.listen(8000);
soap.listen(server, '/soap', service, wsdl);
```

### When to Choose SOAP

**✅ Choose SOAP when:**
- Enterprise systems requiring strict standards
- Legacy system integration (many old systems use SOAP)
- Built-in security requirements (WS-Security)
- ACID transactions needed (WS-AtomicTransaction)
- Formal contracts required (WSDL)
- Financial/banking systems (compliance requirements)

**❌ Avoid SOAP when:**
- Building new public APIs (REST/GraphQL is better)
- Performance is critical (XML is verbose)
- Mobile clients (large payloads, complex parsing)
- Agile development (WSDL changes are rigid)
- Modern web applications (REST is simpler)

### SOAP Pros and Cons

**Pros:**
- ✅ Enterprise features (security, transactions, reliability)
- ✅ Strict standards (WS-* specifications)
- ✅ Language/platform neutral
- ✅ Built-in error handling (SOAP faults)
- ✅ Formal contract (WSDL auto-generates clients)
- ✅ Transport agnostic (HTTP, SMTP, JMS)

**Cons:**
- ❌ Extremely verbose (XML overhead)
- ❌ Complex (steep learning curve)
- ❌ Poor performance (large payloads, slow parsing)
- ❌ Rigid (hard to change contracts)
- ❌ Limited browser support
- ❌ Legacy technology (declining use)

**Recommendation:** Only use SOAP when integrating with existing SOAP services or when enterprise WS-* features are required. For new projects, choose REST, GraphQL, or gRPC.

## Alternative 4: WebSocket APIs

### What are WebSockets?

**WebSocket** provides full-duplex (bidirectional) communication over a single TCP connection. Unlike REST (request-response), both client and server can send messages anytime.

**Core principle:** Persistent connection for real-time bidirectional communication.

### The REST Problem WebSockets Solve

**REST polling (inefficient):**
```javascript
// Problem: Chat app needs real-time messages
// REST approach: Polling (repeatedly ask server for updates)

// Client polls every 2 seconds
setInterval(async () => {
  const response = await fetch('/api/messages/new');
  const newMessages = await response.json();
  
  if (newMessages.length > 0) {
    displayMessages(newMessages);
  }
}, 2000);

// Problems:
// ❌ Wasteful: 30 requests/minute even with no new data
// ❌ Latency: Up to 2 second delay for new messages
// ❌ Server load: Thousands of clients = thousands of requests/second
// ❌ Battery drain: Constant HTTP requests on mobile
```

**WebSocket solution (efficient):**
```javascript
// Persistent connection - server pushes updates instantly

const ws = new WebSocket('ws://localhost:8080');

// Connection opened
ws.onopen = () => {
  console.log('Connected');
};

// Receive messages from server
ws.onmessage = (event) => {
  const message = JSON.parse(event.data);
  displayMessage(message);
  // ✅ Instant delivery (no polling)
};

// Send messages to server
function sendMessage(text) {
  ws.send(JSON.stringify({
    type: 'message',
    text: text
  }));
}

// Benefits:
// ✅ Real-time: Instant message delivery
// ✅ Efficient: One persistent connection
// ✅ Bidirectional: Client and server can both initiate
// ✅ Low latency: No HTTP overhead per message
```

### How WebSockets Work

**Upgrade from HTTP to WebSocket:**
```
Client                           Server
  │                                │
  │   HTTP Upgrade Request         │
  ├────────────────────────────────>│
  │   GET /chat HTTP/1.1           │
  │   Upgrade: websocket           │
  │   Connection: Upgrade          │
  │                                │
  │   HTTP 101 Switching Protocols │
  │<────────────────────────────────┤
  │   Upgrade: websocket           │
  │   Connection: Upgrade          │
  │                                │
  │ ---- WebSocket Connection ---- │ (persistent, bidirectional)
  │<──────────────────────────────>│
  │   Message 1                    │
  │   Message 2                    │
  │<──────────────────────────────>│
```

### Setting Up WebSocket Server (Node.js)

```javascript
// Using 'ws' library
const WebSocket = require('ws');
const http = require('http');

// Create HTTP server
const server = http.createServer();

// Create WebSocket server
const wss = new WebSocket.Server({ server });

// Store connected clients
const clients = new Set();

// Handle connections
wss.on('connection', (ws, req) => {
  console.log('New client connected');
  clients.add(ws);
  
  // Send welcome message
  ws.send(JSON.stringify({
    type: 'welcome',
    message: 'Connected to chat server',
    timestamp: new Date().toISOString()
  }));
  
  // Handle incoming messages
  ws.on('message', (data) => {
    try {
      const message = JSON.parse(data);
      
      // Handle different message types
      switch (message.type) {
        case 'chat':
          handleChatMessage(ws, message);
          break;
        case 'typing':
          handleTypingIndicator(ws, message);
          break;
        case 'ping':
          ws.send(JSON.stringify({ type: 'pong' }));
          break;
        default:
          console.log('Unknown message type:', message.type);
      }
    } catch (error) {
      console.error('Error parsing message:', error);
    }
  });
  
  // Handle disconnection
  ws.on('close', () => {
    console.log('Client disconnected');
    clients.delete(ws);
  });
  
  // Handle errors
  ws.on('error', (error) => {
    console.error('WebSocket error:', error);
  });
});

// Broadcast message to all clients
function broadcast(message) {
  const data = JSON.stringify(message);
  
  clients.forEach((client) => {
    if (client.readyState === WebSocket.OPEN) {
      client.send(data);
    }
  });
}

// Handle chat message
function handleChatMessage(ws, message) {
  // Broadcast to all clients
  broadcast({
    type: 'chat',
    user: message.user,
    text: message.text,
    timestamp: new Date().toISOString()
  });
  
  // Save to database
  db.messages.create({
    user: message.user,
    text: message.text
  });
}

// Handle typing indicator
function handleTypingIndicator(ws, message) {
  // Broadcast to all clients except sender
  clients.forEach((client) => {
    if (client !== ws && client.readyState === WebSocket.OPEN) {
      client.send(JSON.stringify({
        type: 'typing',
        user: message.user,
        isTyping: message.isTyping
      }));
    }
  });
}

// Start server
server.listen(8080, () => {
  console.log('WebSocket server running on port 8080');
});
```

### WebSocket vs REST vs SSE (Server-Sent Events)

| Feature | REST | WebSocket | SSE |
|---------|------|-----------|-----|
| **Direction** | Client → Server | Bidirectional | Server → Client |
| **Connection** | Request/response | Persistent | Persistent |
| **Overhead** | High (HTTP per request) | Low (one connection) | Low (one connection) |
| **Real-time** | No (polling required) | Yes | Yes (server push) |
| **Browser Support** | All browsers | All modern browsers | All browsers |
| **Use Case** | CRUD operations | Chat, gaming, collaboration | Live feeds, notifications |
| **Complexity** | Low | Medium | Low |

**Recommendation:**
- **REST**: Standard CRUD operations
- **WebSocket**: Bidirectional real-time (chat, gaming)
- **SSE**: Server-to-client real-time (notifications, live feeds)

### When to Choose WebSocket

**✅ Choose WebSocket when:**
- Real-time bidirectional communication (chat, gaming)
- Low latency critical (< 100ms)
- High-frequency updates (stock tickers, live sports)
- Server needs to push to clients (notifications)
- Collaborative applications (shared editing)
- Reducing server load from polling

**❌ Avoid WebSocket when:**
- Simple request/response (REST is simpler)
- Infrequent updates (polling is fine)
- One-way server → client (SSE is simpler)
- Need HTTP caching (WebSocket doesn't cache)
- Firewalls/proxies might block (some corporate networks block WS)

### WebSocket Pros and Cons

**Pros:**
- ✅ Real-time bidirectional communication
- ✅ Low latency (no HTTP overhead)
- ✅ Efficient (one persistent connection)
- ✅ Reduced server load (vs polling)
- ✅ Good browser support
- ✅ Can send binary data

**Cons:**
- ❌ More complex than REST (connection management)
- ❌ Scaling is harder (stateful connections)
- ❌ No automatic reconnection (need to implement)
- ❌ Some proxies/firewalls block WebSocket
- ❌ No HTTP caching
- ❌ Harder to debug than HTTP

## Migration Strategies

### Moving from REST to GraphQL

```javascript
// Gradual migration approach

// 1. Keep REST endpoints running
app.get('/api/users/:id', getUserHandler);
app.get('/api/posts', getPostsHandler);

// 2. Add GraphQL alongside
app.use('/graphql', graphqlHTTP({ schema, rootValue: resolvers }));

// 3. GraphQL resolvers can call existing REST logic
const resolvers = {
  user: ({ id }) => getUserHandler({ params: { id } }),
  posts: () => getPostsHandler({})
};

// 4. Gradually move clients to GraphQL
// 5. Deprecate REST endpoints when safe
```

### Moving from REST to gRPC

```javascript
// Strategy: Adapter pattern

// 1. Implement gRPC service
// 2. Create REST → gRPC adapter for backward compatibility

app.post('/api/users', async (req, res) => {
  // Translate REST to gRPC call
  grpcClient.CreateUser(req.body, (error, response) => {
    if (error) {
      return res.status(500).json({ error: error.message });
    }
    res.status(201).json(response);
  });
});

// 3. Internal services use gRPC directly
// 4. Public API continues to use REST adapter
```

**Start with this decision matrix:**

```
Is it a public API?

└─ Yes → gRPC (performance) or REST (simplicity)
```
