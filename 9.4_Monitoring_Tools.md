# 9.4 Monitoring Tools

## The Problem

You've built and deployed your REST API. Users are making requests. Everything seems fine... until it isn't.

**Common scenarios where monitoring saves you:**

1. **Silent failures**: 5% of authentication requests are failing, but no one reported it yet. You're losing users.
2. **Performance degradation**: Response times slowly increased from 100ms to 2 seconds over the past week. What changed?
3. **Capacity planning**: Traffic is growing. When will you need to scale up? 
4. **Debugging production**: A user reports an error at 2:47 PM yesterday. What exactly happened?
5. **Business metrics**: How many API calls per customer? Which endpoints are most used?

**Without monitoring, you're flying blind.** You only learn about problems when users complain, and you have no data to debug with.

**With monitoring, you:**
- Detect issues before users notice
- Understand performance trends over time
- Debug production problems with real data
- Make informed scaling decisions
- Track business metrics alongside technical metrics

This guide covers four monitoring approaches:
1. **New Relic** - Commercial all-in-one APM (Application Performance Monitoring)
2. **Datadog** - Commercial monitoring with excellent integrations
3. **Prometheus + Grafana** - Open-source metrics and visualization
4. **ELK Stack** - Open-source log aggregation and search

---

## Understanding Monitoring Layers

Before diving into tools, understand the **four layers of monitoring**:

### 1. Metrics (Numbers Over Time)
- Request rate (requests per second)
- Response time (p50, p95, p99 percentiles)
- Error rate (percentage of failed requests)
- Resource usage (CPU, memory, disk)

**Example**: "Average response time increased from 150ms to 800ms in the last hour"

### 2. Logs (Event Records)
- Individual request details
- Error messages and stack traces
- Application events (user login, payment processed)

**Example**: "2024-01-15 14:32:18 ERROR - Database connection timeout for user_id=12345"

### 3. Traces (Request Journey)
- Follow a single request through your entire system
- See which services it touched
- Identify where time was spent

**Example**: "Request took 2.3s total: 0.1s in API gateway, 1.8s in database query, 0.4s in payment service"

### 4. Alerts (Proactive Notifications)
- Notify you when something goes wrong
- Trigger based on metrics or logs
- Reduce mean time to detection (MTTD)

**Example**: "ALERT: Error rate exceeded 5% threshold - 127 errors in last 5 minutes"

Most monitoring tools cover multiple layers. Let's explore each tool.

## Tool 1: New Relic (Commercial APM)

**New Relic** is a comprehensive Application Performance Monitoring (APM) platform that automatically instruments your application to collect metrics, traces, and errors.

### What Makes New Relic Special

- **Automatic instrumentation**: Just install the agent, and it captures everything
- **Distributed tracing**: See requests flow through microservices
- **AI-powered insights**: Automatically detects anomalies and suggests fixes
- **Real user monitoring (RUM)**: Track frontend performance too

### Setting Up New Relic (Node.js Example)

```javascript
// Step 1: Install the New Relic agent
// npm install newrelic --save

// Step 2: Create newrelic.js configuration file in your project root
// newrelic.js
exports.config = {
  app_name: ['My REST API'],
  license_key: 'your-license-key-here',
  
  // Logging configuration
  logging: {
    level: 'info',
    filepath: 'stdout'
  },
  
  // Transaction tracer captures slow transactions
  transaction_tracer: {
    enabled: true,
    transaction_threshold: 0.5, // Trace transactions slower than 500ms
    record_sql: 'obfuscated',   // Don't log actual query parameters
    explain_threshold: 500      // Get query plans for slow queries
  },
  
  // Error collector
  error_collector: {
    enabled: true,
    ignore_status_codes: [404], // Don't alert on expected 404s
    expected_messages: {
      'ValidationError': ['Invalid email format'] // Expected business errors
    }
  },
  
  // Custom instrumentation
  custom_instrumentation_editor: {
    enabled: true
  },
  
  // Distributed tracing for microservices
  distributed_tracing: {
    enabled: true
  }
};
```

**Explanation of configuration:**
- `transaction_threshold`: Only trace requests slower than this (avoid overhead)
- `record_sql: 'obfuscated'`: Log SQL queries but hide sensitive data
- `ignore_status_codes`: Don't treat certain status codes as errors
- `distributed_tracing`: Essential for microservices to track requests across services

```javascript
// Step 3: Require newrelic as the FIRST line of your app
// server.js
require('newrelic'); // Must be first!

const express = require('express');
const app = express();

// Your existing code continues normally...
app.use(express.json());

// New Relic automatically tracks these routes
app.get('/api/users', async (req, res) => {
  // New Relic automatically tracks:
  // - Response time
  // - Status code
  // - Database queries
  // - External API calls
  const users = await db.query('SELECT * FROM users');
  res.json(users);
});

app.post('/api/orders', async (req, res) => {
  try {
    const order = await createOrder(req.body);
    res.status(201).json(order);
  } catch (error) {
    // New Relic automatically captures this error with full context
    res.status(500).json({ error: 'Order creation failed' });
  }
});

app.listen(3000);
```

**How it works:**
- New Relic monkey-patches Node.js to intercept function calls
- Automatically tracks HTTP requests, database queries, external API calls
- No code changes needed for basic monitoring
- Errors are captured with full stack traces and context

### Adding Custom Metrics

Sometimes you need to track business-specific metrics beyond the basics:

```javascript
const newrelic = require('newrelic');

app.post('/api/checkout', async (req, res) => {
  const startTime = Date.now();
  
  try {
    // Process payment
    const payment = await processPayment(req.body);
    
    // Track custom metric: successful payment amount
    newrelic.recordMetric('Custom/PaymentAmount', payment.amount);
    
    // Track custom event with attributes
    newrelic.recordCustomEvent('PaymentProcessed', {
      userId: req.user.id,
      amount: payment.amount,
      currency: payment.currency,
      paymentMethod: payment.method,
      processingTime: Date.now() - startTime
    });
    
    res.json({ success: true, orderId: payment.orderId });
    
  } catch (error) {
    // Add custom attributes to error for better debugging
    newrelic.noticeError(error, {
      userId: req.user.id,
      cartTotal: req.body.amount,
      paymentMethod: req.body.method
    });
    
    res.status(500).json({ error: 'Payment failed' });
  }
});
```

**Why custom metrics matter:**
- `recordMetric`: Track numeric business metrics (revenue, conversions, etc.)
- `recordCustomEvent`: Track discrete events with rich context
- `noticeError`: Manually report errors with additional context
- These appear in New Relic dashboards alongside technical metrics

### Creating Custom Segments (Detailed Timing)

```javascript
const newrelic = require('newrelic');

app.get('/api/recommendations/:userId', async (req, res) => {
  // Track time spent in each part of the request
  
  // Segment 1: Fetch user profile
  const userProfile = await newrelic.startSegment(
    'Fetch User Profile',
    true, // Record as database operation
    async () => {
      return await db.users.findById(req.params.userId);
    }
  );
  
  // Segment 2: Calculate recommendations (external ML service)
  const recommendations = await newrelic.startSegment(
    'Calculate Recommendations',
    false, // Not a database operation
    async () => {
      return await mlService.getRecommendations(userProfile);
    }
  );
  
  // Segment 3: Fetch product details
  const products = await newrelic.startSegment(
    'Fetch Product Details',
    true,
    async () => {
      return await db.products.findByIds(recommendations.map(r => r.productId));
    }
  );
  
  res.json(products);
});
```

**How segments help:**
- New Relic shows you: "This request took 2.3s: 0.1s fetching user, 1.8s calling ML service, 0.4s fetching products"
- Identifies bottlenecks: "ML service is the slow part"
- Tracks external dependencies separately from your code

### New Relic Dashboard Example

```javascript
// You can query New Relic data using NRQL (New Relic Query Language)
// Similar to SQL but for metrics and events

// Query 1: Average response time by endpoint
SELECT average(duration) 
FROM Transaction 
WHERE appName = 'My REST API'
FACET name 
SINCE 1 hour ago

// Query 2: Error rate over time
SELECT percentage(count(*), WHERE error IS true) 
FROM Transaction 
SINCE 1 day ago 
TIMESERIES

// Query 3: Slowest database queries
SELECT average(duration), query 
FROM Datastore 
WHERE appName = 'My REST API'
FACET query 
SINCE 1 hour ago 
LIMIT 10

// Query 4: Custom business metric - Revenue by payment method
SELECT sum(amount) 
FROM PaymentProcessed 
FACET paymentMethod 
SINCE 1 day ago

// Query 5: 95th percentile response time (what 95% of users experience)
SELECT percentile(duration, 95) 
FROM Transaction 
SINCE 1 hour ago 
TIMESERIES
```

**Why these queries matter:**
- **Average response time by endpoint**: Find your slowest endpoints
- **Error rate over time**: See if errors are increasing
- **Slowest database queries**: Identify queries that need optimization
- **Business metrics**: Connect technical performance to business outcomes
- **Percentiles**: Average can be misleading (one slow request drags it up); p95 shows what most users experience

### Pros and Cons of New Relic

**Pros:**
- ✅ Zero-code setup for basic monitoring
- ✅ Automatic distributed tracing across microservices
- ✅ AI-powered anomaly detection
- ✅ Excellent out-of-the-box dashboards
- ✅ Real user monitoring (track frontend too)
- ✅ Great for teams without monitoring expertise

**Cons:**
- ❌ Expensive for high-traffic APIs ($0.10-0.30 per GB ingested)
- ❌ Vendor lock-in (hard to migrate away)
- ❌ Less customizable than open-source solutions
- ❌ Can't run on-premises (cloud-only SaaS)

**When to choose New Relic:**
- Your team values speed over cost
- You need monitoring up and running quickly
- You want automatic instrumentation
- You don't have dedicated DevOps to manage infrastructure

**When to avoid:**
- Budget is tight (open-source alternatives exist)
- You need on-premises deployment
- You require highly customized monitoring

## Tool 2: Datadog (Commercial Monitoring Platform)

**Datadog** is a cloud monitoring platform that unifies metrics, traces, and logs in one place. It's known for its extensive integrations and beautiful visualizations.

### What Makes Datadog Special

- **700+ integrations**: Monitors everything (databases, caches, queues, cloud services)
- **Unified platform**: Metrics, logs, and traces in one UI
- **Custom metrics**: Easy to send any metric you want
- **Anomaly detection**: ML-powered alerts
- **Log management**: Powerful log search and analysis

### Setting Up Datadog (Node.js Example)

```javascript
// Step 1: Install Datadog tracer
// npm install dd-trace --save

// Step 2: Initialize at the start of your application
// server.js
const tracer = require('dd-trace').init({
  // Configuration
  service: 'my-api',           // Service name in Datadog
  env: 'production',            // Environment tag
  version: '1.2.3',             // Version for deployment tracking
  
  // Logging
  logInjection: true,           // Add trace IDs to logs
  
  // Performance
  runtimeMetrics: true,         // Collect Node.js runtime metrics
  
  // Tracing
  analytics: true,              // Enable trace analytics
  
  // Sampling (reduce cost for high traffic)
  sampleRate: 1.0,              // 1.0 = trace 100% of requests
                                 // 0.1 = trace 10% (saves costs)
  
  // Tags applied to all metrics
  tags: {
    team: 'backend',
    region: 'us-east-1'
  }
});

const express = require('express');
const app = express();

app.use(express.json());

// Datadog automatically instruments Express routes
app.get('/api/products', async (req, res) => {
  // Automatically tracked:
  // - Request rate
  // - Response time
  // - Status codes
  const products = await db.query('SELECT * FROM products');
  res.json(products);
});

app.listen(3000);
```

**Configuration explained:**
- `service`: Groups all metrics/traces from this app
- `logInjection`: Correlates logs with traces (see which log came from which request)
- `runtimeMetrics`: Tracks Node.js garbage collection, event loop lag, memory usage
- `sampleRate`: For high-traffic APIs, trace a sample to reduce costs

### Sending Custom Metrics to Datadog

```javascript
const tracer = require('dd-trace');
const StatsD = require('hot-shots'); // Datadog's StatsD client

// Initialize StatsD client
const dogstatsd = new StatsD({
  host: 'localhost',
  port: 8125,
  prefix: 'myapi.',      // Prefix all metrics with "myapi."
  globalTags: {
    env: 'production',
    service: 'my-api'
  }
});

app.post('/api/orders', async (req, res) => {
  const startTime = Date.now();
  
  try {
    const order = await createOrder(req.body);
    
    // Increment counter: total orders
    dogstatsd.increment('orders.created', 1, {
      product_type: order.productType,
      payment_method: order.paymentMethod
    });
    
    // Track order value
    dogstatsd.histogram('orders.value', order.total, {
      currency: order.currency
    });
    
    // Track processing time
    dogstatsd.timing('orders.processing_time', Date.now() - startTime);
    
    res.status(201).json(order);
    
  } catch (error) {
    // Increment error counter
    dogstatsd.increment('orders.errors', 1, {
      error_type: error.name
    });
    
    res.status(500).json({ error: 'Order creation failed' });
  }
});
```

**Metric types explained:**
- **Counter** (`increment`): Counts events (orders created, errors occurred)
- **Histogram**: Measures distribution (order values, response times) - Datadog calculates avg, p95, p99
- **Timing**: Measures durations (how long operations take)
- **Gauge**: Measures current value (current connections, queue depth)

### Adding Tracing Spans

```javascript
const tracer = require('dd-trace');

app.get('/api/recommendations/:userId', async (req, res) => {
  // Create custom span to measure specific operation
  const span = tracer.startSpan('fetch.user.recommendations');
  
  // Add metadata to span
  span.setTag('user.id', req.params.userId);
  span.setTag('user.tier', req.user.tier);
  
  try {
    // Step 1: Fetch user preferences
    const preferences = await tracer.trace('fetch.user.preferences', async () => {
      return await db.preferences.find({ userId: req.params.userId });
    });
    
    // Step 2: Call recommendation engine
    const recommendations = await tracer.trace('recommendation.engine', async () => {
      const result = await mlService.getRecommendations({
        userId: req.params.userId,
        preferences: preferences
      });
      
      // Add info about the result
      tracer.scope().active().setTag('recommendations.count', result.length);
      
      return result;
    });
    
    // Step 3: Hydrate with product data
    const products = await tracer.trace('fetch.product.details', async () => {
      return await db.products.findByIds(recommendations.map(r => r.productId));
    });
    
    span.setTag('result.count', products.length);
    span.finish(); // Mark span as complete
    
    res.json(products);
    
  } catch (error) {
    span.setTag('error', true);
    span.setTag('error.message', error.message);
    span.finish();
    throw error;
  }
});
```

**Why custom spans help:**
- See exactly where time is spent: "2.3s total: 0.1s preferences, 2.0s recommendation engine, 0.2s products"
- Track metadata: Which user tier? How many results?
- Debug slow requests: "Recommendation engine was slow for VIP users"

## Tool 3: Prometheus + Grafana (Open-Source Metrics)

**Prometheus** collects and stores metrics. **Grafana** visualizes them. Together, they're the most popular open-source monitoring stack.

### What Makes Prometheus + Grafana Special

- **Free and open-source**: No licensing costs
- **Self-hosted**: Full control over your data
- **Pull-based**: Prometheus scrapes metrics from your app (no agent needed)
- **Powerful queries**: PromQL for flexible metric analysis
- **Active community**: Thousands of exporters and integrations

### Architecture Overview

```
┌─────────────┐
│  Your API   │ ← Exposes /metrics endpoint
└──────┬──────┘
       │
       │ HTTP GET /metrics every 15s
       ↓
┌─────────────┐
│ Prometheus  │ ← Scrapes and stores metrics
└──────┬──────┘
       │
       │ Query metrics
       ↓
┌─────────────┐
│  Grafana    │ ← Visualizes metrics
└─────────────┘
```

**How it works:**
1. Your API exposes a `/metrics` endpoint with current metrics
2. Prometheus scrapes this endpoint every 15 seconds
3. Prometheus stores time-series data
4. Grafana queries Prometheus and displays dashboards

### Setting Up Prometheus Metrics in Node.js

```javascript
// Step 1: Install Prometheus client
// npm install prom-client --save

const express = require('express');
const promClient = require('prom-client');

const app = express();
app.use(express.json());

// Create a Registry (holds all metrics)
const register = new promClient.Registry();

// Add default metrics (CPU, memory, garbage collection)
promClient.collectDefaultMetrics({ register });

// Create custom metrics

// Counter: Counts events (always increases)
const httpRequestsTotal = new promClient.Counter({
  name: 'http_requests_total',
  help: 'Total number of HTTP requests',
  labelNames: ['method', 'route', 'status_code'], // Dimensions
  registers: [register]
});

// Histogram: Measures distribution (response times)
const httpRequestDuration = new promClient.Histogram({
  name: 'http_request_duration_seconds',
  help: 'Duration of HTTP requests in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.1, 0.5, 1, 2, 5, 10], // Measure: <100ms, <500ms, <1s, etc.
  registers: [register]
});

// Gauge: Current value (goes up and down)
const activeConnections = new promClient.Gauge({
  name: 'active_connections',
  help: 'Number of active connections',
  registers: [register]
});

// Summary: Like histogram, but calculates percentiles
const requestSize = new promClient.Summary({
  name: 'http_request_size_bytes',
  help: 'Size of HTTP requests in bytes',
  labelNames: ['method', 'route'],
  percentiles: [0.5, 0.9, 0.95, 0.99], // p50, p90, p95, p99
  registers: [register]
});

// Middleware to track all requests
app.use((req, res, next) => {
  activeConnections.inc(); // Increment active connections
  
  const start = Date.now();
  
  // Track request size
  if (req.headers['content-length']) {
    requestSize.labels(req.method, req.route?.path || req.path)
               .observe(parseInt(req.headers['content-length']));
  }
  
  // When response finishes
  res.on('finish', () => {
    const duration = (Date.now() - start) / 1000; // Convert to seconds
    const route = req.route?.path || req.path;
    
    // Record metrics
    httpRequestsTotal.labels(req.method, route, res.statusCode).inc();
    httpRequestDuration.labels(req.method, route, res.statusCode).observe(duration);
    activeConnections.dec(); // Decrement active connections
  });
  
  next();
});

// Your API routes
app.get('/api/users', async (req, res) => {
  const users = await db.query('SELECT * FROM users');
  res.json(users);
});

app.post('/api/users', async (req, res) => {
  const user = await db.insert('users', req.body);
  res.status(201).json(user);
});

// Expose metrics endpoint for Prometheus to scrape
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});

app.listen(3000);
```

**Metric types explained:**
- **Counter**: Total requests, total errors (only goes up)
- **Histogram**: Response time distribution (Prometheus calculates p50, p95, p99)
- **Gauge**: Current active connections, current queue size (goes up and down)
- **Summary**: Like histogram, but application calculates percentiles (more CPU, less storage)

**Labels (dimensions):**
- `labelNames: ['method', 'route']` lets you query: "Show me response time for GET /api/users"
- Without labels, you'd only know overall response time for all endpoints combined

    ordersCreated.labels(order.productType, order.paymentMethod).inc();
    
    // Track order value
    orderValue.labels(order.currency).observe(order.total);
    
    res.status(201).json(order);
  } catch (error) {
    res.status(500).json({ error: 'Order creation failed' });
  }
});

// Update subscription count periodically
setInterval(async () => {
  const counts = await db.query(`
    SELECT plan_type, COUNT(*) as count 
    FROM subscriptions 
    WHERE status = 'active'
    GROUP BY plan_type
  `);
  
  counts.forEach(row => {
    activeSubscriptions.labels(row.plan_type).set(row.count);
  });
}, 60000); // Every minute

### Writing PromQL Queries

PromQL is Prometheus's query language. Here are common queries:

```promql
# Query 1: Current request rate (requests per second)
rate(http_requests_total[5m])

# Explanation: 
# - http_requests_total is a counter (always increasing)
# - rate() calculates per-second increase over last 5 minutes
# - Result: "We're receiving 150 requests/second"

# Query 2: Request rate by endpoint
sum(rate(http_requests_total[5m])) by (route)

# Explanation:
# - sum() aggregates across all label combinations
# - by (route) groups by endpoint
# - Result: "GET /api/users: 50 req/s, POST /api/orders: 30 req/s"

# Query 3: Error rate percentage
(
  sum(rate(http_requests_total{status_code=~"5.."}[5m]))
  /
  sum(rate(http_requests_total[5m]))
) * 100

# Explanation:
# - top: errors (status codes 5xx)
# - bottom: all requests
# - multiply by 100 for percentage
# - Result: "2.3% of requests are failing"

# Query 4: 95th percentile response time
histogram_quantile(0.95, 
  sum(rate(http_request_duration_seconds_bucket[5m])) by (le)
)

# Explanation:
# - histogram_quantile() calculates percentiles from histogram
# - 0.95 = 95th percentile
# - Result: "95% of requests complete in < 0.8 seconds"

# Query 5: 95th percentile by endpoint
histogram_quantile(0.95, 
  sum(rate(http_request_duration_seconds_bucket[5m])) by (route, le)
)

# Explanation:
# - by (route, le) groups by endpoint
# - Result shows p95 for each endpoint separately

# Query 6: Requests per minute (instead of per second)
sum(increase(http_requests_total[1m]))

# Explanation:
# - increase() shows total increase over time window
# - [1m] = last minute
# - Result: "We received 5,420 requests in the last minute"

# Query 7: Average request size
rate(http_request_size_bytes_sum[5m]) 
/ 
rate(http_request_size_bytes_count[5m])

# Explanation:
# - Summaries track _sum and _count
# - sum / count = average
# - Result: "Average request size is 1,245 bytes"

# Query 8: Business metric - Revenue per minute
sum(rate(order_value_dollars_sum[1m])) * 60

# Explanation:
# - order_value_dollars_sum tracks cumulative order value
# - rate() gives dollars per second
# - * 60 converts to dollars per minute
# - Result: "$12,450 revenue/minute"

# Query 9: Memory usage trend (is it growing?)
rate(process_resident_memory_bytes[5m])

# Explanation:
# - positive rate = memory increasing (potential leak)
# - negative rate = memory decreasing
# - Result: "Memory growing at 50KB/second"

# Query 10: Prediction (will we run out of disk?)
predict_linear(node_filesystem_free_bytes[1h], 4*3600)

# Explanation:
# - Fits linear regression to last hour of data
# - Predicts value 4 hours from now (4*3600 seconds)
# - Result: "In 4 hours, we'll have 5.2GB free"
```

### Pros and Cons of Prometheus + Grafana

**Pros:**
- ✅ Free and open-source (no licensing costs)
- ✅ Self-hosted (full control, no vendor lock-in)
- ✅ Powerful query language (PromQL)
- ✅ Excellent for metrics and alerting
- ✅ Large ecosystem (many exporters and integrations)
- ✅ Grafana dashboards are beautiful and highly customizable

**Cons:**
- ❌ Requires infrastructure management (you host it)
- ❌ No built-in log management (need separate tool like Loki)
- ❌ No distributed tracing (need Jaeger or Zipkin)
- ❌ Steeper learning curve (PromQL, configuration)
- ❌ Long-term storage requires additional setup (Prometheus keeps only 15 days by default)

**When to choose Prometheus + Grafana:**
- Cost is a concern (it's free)
- You want full control over your monitoring data
- You need on-premises deployment
- You have DevOps expertise to manage infrastructure
- Metrics and alerting are your primary needs

**When to avoid:**
- Small team without DevOps expertise
- Need unified metrics + logs + traces in one tool
- Want monitoring up and running quickly
- Need managed service (no infrastructure management)

## Tool 4: ELK Stack (Elasticsearch, Logstash, Kibana)

The **ELK Stack** is for log aggregation, search, and analysis. It's the most popular open-source logging solution.

### What Makes ELK Special

- **Elasticsearch**: Stores and indexes logs (search billions of logs in seconds)
- **Logstash**: Collects, transforms, and sends logs to Elasticsearch
- **Kibana**: Visualizes logs and creates log-based dashboards
- **Powerful search**: Full-text search across all logs
- **Log analysis**: Find patterns, trends, anomalies

### Configuring Logstash

```ruby
# logstash.conf - Logstash pipeline

# Input: Receive from Filebeat
input {
  beats {
    port => 5044
  }
}

# Filter: Parse, transform, enrich logs
filter {
  # Already JSON from Filebeat, but let's enrich
  
  # Parse timestamp
  date {
    match => ["timestamp", "ISO8601"]
    target => "@timestamp"
  }
  
  # Add geolocation based on IP
  geoip {
    source => "ip"
    target => "geo"
  }
  
  # Categorize by status code
  if [status] >= 500 {
    mutate {
      add_field => { "severity" => "error" }
    }
  } else if [status] >= 400 {
    mutate {
      add_field => { "severity" => "warning" }
    }
  } else {
    mutate {
      add_field => { "severity" => "info" }
    }
  }
  
  # Parse user agent
  useragent {
    source => "user_agent"
    target => "ua"
  }
  
  # Remove fields we don't need
  mutate {
    remove_field => ["@version", "host"]
  }
}

# Output: Send to Elasticsearch
output {
  elasticsearch {
    hosts => ["localhost:9200"]
    index => "api-logs-%{+YYYY.MM.dd}"  # Daily indices
    
    # Mapping template
    template_name => "api-logs"
  }
  
  # Also output to stdout for debugging
  stdout {
    codec => rubydebug
  }
}
```

**What Logstash does:**
- Parses timestamps
- Adds geolocation (country, city from IP address)
- Categorizes logs by severity
- Parses user agent (browser, OS)
- Sends to Elasticsearch with daily indices

### Pros and Cons of ELK Stack

**Pros:**
- ✅ Free and open-source
- ✅ Powerful log search (search billions of logs in seconds)
- ✅ Excellent for debugging production issues
- ✅ Log analysis and pattern detection
- ✅ Beautiful visualizations in Kibana
- ✅ Self-hosted (full control over logs)

**Cons:**
- ❌ Resource-intensive (Elasticsearch needs lots of RAM)
- ❌ Complex setup (4+ components to configure)
- ❌ Requires significant infrastructure
- ❌ Expensive to scale (storage costs grow quickly)
- ❌ No built-in metrics or tracing (logs only)

**When to choose ELK:**
- Need powerful log search and analysis
- Debugging complex production issues
- Large volume of logs (millions per day)
- Want full control over log data
- Have DevOps expertise to manage infrastructure

**When to avoid:**
- Small team without DevOps expertise
- Low budget (storage costs for large log volumes)
- Need metrics + traces too (ELK is logs only)
- Want quick setup (ELK is complex)

- ✅ You want automatic instrumentation (zero code changes)
- ✅ You value support and documentation

### Choose **Datadog** if:
- ✅ You have complex infrastructure (many services, databases, queues)
- ✅ You want metrics + logs + traces unified in one tool
- ✅ You need 700+ integrations
- ✅ Budget allows for comprehensive monitoring ($1000-5000/month)
- ✅ You want beautiful, pre-built dashboards
- ✅ You value anomaly detection and AI-powered insights

### Choose **Prometheus + Grafana** if:
- ✅ Cost is a primary concern (it's free)
- ✅ Metrics and alerting are your main needs
- ✅ You have DevOps expertise to manage infrastructure
- ✅ You want full control (no vendor lock-in)

### Choose **ELK Stack** if:
- ✅ Log search and analysis are critical

## Best Practices for Monitoring

### 1. Start with the Golden Signals

Monitor these four metrics first (from Google's SRE book):

```javascript
// 1. Latency - How long requests take
// Track: p50, p95, p99 response times

// 2. Traffic - How much demand
// Track: Requests per second

// 3. Errors - Rate of failed requests
// Track: Error count, error percentage

// 4. Saturation - How "full" your system is
// Track: CPU, memory, disk usage, queue depth
```

### 2. Use Structured Logging

```javascript
// ❌ Bad: Unstructured logs
console.log('User logged in');
console.log(`Order created: ${orderId}`);

// ✅ Good: Structured JSON logs
logger.info('User logged in', {
  user_id: userId,
  ip: req.ip,
  user_agent: req.headers['user-agent']
});

logger.info('Order created', {
  order_id: orderId,
  user_id: userId,
  total: orderTotal,
  items_count: items.length
});
```

**Why:** Structured logs are searchable, filterable, and analyzable.

### 3. Tag Everything

```javascript
// Add context to metrics and logs

// Metrics with tags
requestCounter.labels({
  method: req.method,
  route: req.route,
  status: res.statusCode,
  customer_tier: user.tier  // Business context
}).inc();

// Logs with tags
logger.info('Payment processed', {
  user_id: userId,
  amount: amount,
  payment_method: method,
  experiment_group: user.experimentGroup  // A/B test context
});
```

**Why:** Tags let you slice data: "Show errors for VIP customers" or "Compare response time across A/B test groups"

### 4. Set Up Alerts Thoughtfully

```javascript
// ❌ Bad: Alert on everything
// Result: Alert fatigue, ignored alerts

// ✅ Good: Alert on what matters
// - Error rate > 5% (user impact)
// - Response time p95 > 1s (poor experience)
// - Traffic drop > 50% (possible outage)
// - Revenue drop > 30% (business impact)

// Use severity levels:
// - Critical: Page on-call engineer
// - Warning: Notify Slack channel
// - Info: Log for review
```

### 5. Monitor Business Metrics Alongside Technical Metrics

```javascript
// Technical metrics alone don't tell the full story

// ✅ Track both:
// Technical: Response time, error rate, CPU
// Business: Orders placed, revenue, user signups, conversions

// Example: You'd never know from technical metrics that:
// "Payment success rate dropped from 98% to 92%"
// "Average order value decreased by 30%"
```

### 6. Use Percentiles, Not Averages

```javascript
// ❌ Bad: Average response time
// Average: 200ms
// Reality: 90% of requests are 50ms, 10% are 2000ms
// Users experiencing 2s delays, but average looks fine!

// ✅ Good: Use percentiles
// p50 (median): 50ms - typical user experience
// p95: 180ms - what 95% of users experience
// p99: 1800ms - what slowest 1% experience

// If p99 is bad, 1 in 100 requests is slow
// For a high-traffic API, that's thousands of users!
```

### 7. Create Runbooks for Alerts

```javascript
// Every alert should have a runbook

/*
ALERT: HighErrorRate

What: Error rate exceeded 5% threshold
When to worry: If sustained for >5 minutes
Impact: Users experiencing failures

Investigation steps:
1. Check Kibana logs: Search for level:"error" in last 10 minutes
2. Identify error pattern: Database timeouts? External service down?
3. Check dependencies: Is payment service / database healthy?
4. Check metrics: Did traffic spike? Memory leak?

Common causes:
- Database connection pool exhausted → Scale DB
- External service down → Enable circuit breaker
- Deployment bug → Rollback

Resolution:
- If dependency issue: Contact responsible team
- If capacity issue: Scale up
- If code bug: Rollback deployment

Escalation: Page @backend-lead if unresolved in 15 minutes
*/
```

### 8. Monitor Your Monitoring

```javascript
// Your monitoring system can fail too!

// ❌ Don't assume monitoring is always working
// ✅ Do monitor your monitoring:

// 1. Prometheus: Alert if scrape errors
// 2. Filebeat: Alert if log shipping stops
// 3. Datadog Agent: Monitor agent health
// 4. Set up external health checks (e.g., Pingdom, UptimeRobot)

// Synthetic monitoring: Periodically test critical flows
setInterval(async () => {
  try {
    // Simulate user journey
    const loginResult = await api.login(testUser);
    const productResult = await api.getProducts();
    const checkoutResult = await api.checkout(testOrder);
    
    // Report success
    metrics.syntheticCheck.labels('checkout_flow').inc();
  } catch (error) {
    // Report failure
    metrics.syntheticCheckFailures.labels('checkout_flow').inc();
    logger.error('Synthetic check failed', { error });
  }
}, 60000); // Every minute
```

### 9. Set Retention Policies

```javascript
// Logs and metrics accumulate FAST

// Example retention policy:
// - Raw logs: 7 days (for debugging recent issues)
// - Aggregated logs: 30 days (for trend analysis)
// - Error logs: 90 days (for long-term pattern analysis)
// - Metrics (high resolution): 30 days
// - Metrics (downsampled): 1 year

// Why: Balance between storage costs and debugging needs
// Most issues are caught within 7 days
// Historical trends don't need minute-by-minute resolution
```

### 10. Create Dashboards for Different Audiences

```javascript
// Dashboard 1: Engineering - Technical health
// - Request rate, error rate, response time
// - Database query performance
// - Memory/CPU usage
// - Recent error logs

// Dashboard 2: On-Call - Is something broken?
// - Current error rate (big number, red if >5%)
// - Current response time (p95)
// - Traffic compared to normal
// - Recent alerts

// Dashboard 3: Business - Business metrics
// - Orders per hour
// - Revenue per hour
// - Conversion rate
// - Active users

// Dashboard 4: Product - Feature usage
// - API endpoints by traffic
// - Feature adoption rate
// - A/B test metrics
```

## Key Takeaways

1. **You can't fix what you can't see** - Monitoring is essential for production APIs
   
2. **Monitor the four golden signals**: Latency, Traffic, Errors, Saturation
   
3. **Choose tools based on your needs:**
   - Quick setup + premium features → New Relic or Datadog
   - Cost-conscious + metrics focus → Prometheus + Grafana
   - Log search + debugging → ELK Stack
   - Complete open-source stack → Prometheus + Loki + Jaeger

4. **Start simple, add complexity as needed** - Don't over-engineer monitoring day one

5. **Structure your logs** - JSON logs are searchable and analyzable

6. **Use tags everywhere** - Enable slicing and dicing of data

7. **Alert on impact, not symptoms** - Alert when users are affected, not on every metric anomaly

8. **Monitor business metrics** - Technical metrics don't show the full picture

9. **Create runbooks** - Every alert needs investigation steps

10. **Monitoring is an ongoing process** - Regularly review and refine
