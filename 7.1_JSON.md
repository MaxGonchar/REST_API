# JSON - JavaScript Object Notation

## Introduction

**JSON (JavaScript Object Notation)** is the dominant data format for REST APIs. It's lightweight, human-readable, and universally supported.

**Key principle:** JSON is simple, yet powerful enough for 99% of API use cases.

---

## What is JSON?

JSON is a text-based data interchange format that represents structured data using JavaScript object syntax.

```json
{
  "name": "John Doe",
  "age": 30,
  "email": "john@example.com",
  "active": true,
  "roles": ["user", "admin"],
  "address": {
    "street": "123 Main St",
    "city": "New York"
  }
}
```

---

## JSON Data Types

### 1. String

```json
{
  "name": "John Doe",
  "description": "Software developer with 5+ years experience",
  "emoji": "🚀"
}
```

**Rules:**
- Must be wrapped in double quotes `"`
- Escape special characters: `\"`, `\\`, `\/`, `\b`, `\f`, `\n`, `\r`, `\t`
- Supports Unicode: `\u0041` = "A"

### 2. Number

```json
{
  "age": 30,
  "price": 19.99,
  "quantity": 100,
  "discount": 0.15,
  "scientific": 1.2e5,
  "negative": -42
}
```

**Rules:**
- No quotes around numbers
- Can be integer or floating-point
- Can use scientific notation
- No leading zeros: `042` is invalid, use `42`
- No `NaN` or `Infinity` (not valid JSON)

### 3. Boolean

```json
{
  "active": true,
  "verified": false,
  "deleted": false
}
```

**Rules:**
- Only `true` or `false` (lowercase)
- No quotes

### 4. Null

```json
{
  "middleName": null,
  "deletedAt": null
}
```

**Rules:**
- Represents absence of value
- Lowercase `null` only

### 5. Array

```json
{
  "tags": ["javascript", "nodejs", "api"],
  "scores": [95, 87, 92, 88],
  "mixed": [1, "text", true, null],
  "nested": [
    [1, 2, 3],
    [4, 5, 6]
  ]
}
```

**Rules:**
- Ordered collection
- Wrapped in square brackets `[]`
- Elements separated by commas
- Can contain any JSON type
- Can be nested

### 6. Object

```json
{
  "user": {
    "id": 42,
    "name": "John Doe",
    "settings": {
      "theme": "dark",
      "notifications": true
    }
  }
}
```

**Rules:**
- Unordered collection of key-value pairs
- Keys must be strings in double quotes
- Values can be any JSON type
- Wrapped in curly braces `{}`
- Can be nested

## JSON Syntax Rules

### Valid JSON

```json
{
  "name": "John Doe",
  "age": 30,
  "hobbies": ["reading", "coding"],
  "address": {
    "city": "New York"
  }
}
```

### Common Syntax Errors

```json
// ❌ INVALID: Single quotes
{ 'name': 'John' }

// ✅ VALID: Double quotes
{ "name": "John" }

// ❌ INVALID: Trailing comma
{ "name": "John", }

// ✅ VALID: No trailing comma
{ "name": "John" }

// ❌ INVALID: Unquoted keys
{ name: "John" }

// ✅ VALID: Quoted keys
{ "name": "John" }

// ❌ INVALID: Comments (not allowed in JSON)
{
  // This is a comment
  "name": "John"
}

// ✅ VALID: No comments
{
  "name": "John"
}

// ❌ INVALID: Undefined, NaN, Infinity
{ "value": undefined }
{ "calc": NaN }
{ "max": Infinity }

// ✅ VALID: Use null or strings
{ "value": null }
{ "calc": null }
{ "max": "Infinity" }

// ❌ INVALID: Functions, dates as objects
{ "func": function() {} }
{ "date": new Date() }

// ✅ VALID: Dates as ISO strings
{ "date": "2026-03-07T10:00:00Z" }
```

## Pros and Cons of JSON

### ✅ Advantages (Pros)

#### 1. **Human-Readable**

```json
{
  "user": {
    "name": "John Doe",
    "email": "john@example.com"
  }
}
```

Easy to read and understand without specialized tools.

#### 2. **Lightweight**

```json
// JSON: 45 bytes
{"id":1,"name":"John","age":30}
```

Compared to XML (more verbose):
```xml
<!-- XML: 78 bytes -->
<user>
  <id>1</id>
  <name>John</name>
  <age>30</age>
</user>
```

**JSON is ~40% smaller than equivalent XML**

#### 3. **Language-Independent**

JSON parsers exist for every programming language:

```javascript
// JavaScript
const data = JSON.parse(jsonString);

// Python
import json
data = json.loads(json_string)

// Java
JSONObject obj = new JSONObject(jsonString);

// Go
json.Unmarshal([]byte(jsonStr), &data)

// PHP
$data = json_decode($json_string);
```

#### 4. **Native JavaScript Support**

```javascript
// Parse JSON
const obj = JSON.parse('{"name":"John"}');
console.log(obj.name); // "John"

// Stringify object
const json = JSON.stringify({ name: "John" });
console.log(json); // '{"name":"John"}'
```

No external library needed in JavaScript!

#### 5. **Fast Parsing**

JSON parsers are highly optimized:
- **Parsing 1MB JSON:** ~5ms
- **Parsing 1MB XML:** ~50ms

**JSON is 10x faster to parse than XML**

#### 6. **Simple Structure**

Only 6 data types:
- String
- Number
- Boolean
- Null
- Array
- Object

Easy to learn and implement.

#### 7. **Direct Mapping to Data Structures**

```javascript
// JSON maps directly to JavaScript objects
{
  "user": {
    "name": "John",
    "hobbies": ["reading", "coding"]
  }
}

// Becomes:
const user = {
  user: {
    name: "John",
    hobbies: ["reading", "coding"]
  }
};
```

#### 8. **Wide Adoption**

Used by virtually every REST API:
- Twitter API
- GitHub API
- Stripe API
- Google APIs
- Facebook Graph API

#### 9. **Schema Validation**

JSON Schema provides validation:

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "name": { "type": "string" },
    "age": { "type": "number", "minimum": 0 }
  },
  "required": ["name"]
}
```

#### 10. **Compact for Data Transfer**

Perfect for APIs where bandwidth matters:
- Mobile apps (limited data)
- Real-time applications
- High-traffic APIs

---

### ❌ Disadvantages (Cons)

#### 1. **No Comments**

```json
{
  "name": "John",
  "age": 30
}
```

Cannot add explanatory comments in JSON files. Must use external documentation.

**Workaround:**
```json
{
  "_comment": "This is a user object",
  "name": "John",
  "age": 30
}
```

But this is not a standard practice.

#### 2. **No Date Type**

```json
{
  "created_at": "2026-03-07T10:00:00Z"
}
```

Dates must be stored as strings (ISO 8601 format recommended).

**Parsing required:**
```javascript
const date = new Date(obj.created_at);
```

#### 3. **Limited Number Precision**

JavaScript numbers are IEEE 754 double-precision:
- Maximum safe integer: `2^53 - 1` (9,007,199,254,740,991)
- Beyond this, precision is lost

```javascript
// ❌ Problem: Large numbers lose precision
const largeId = 9007199254740993;
JSON.parse(JSON.stringify(largeId)); // 9007199254740992 (wrong!)

// ✅ Solution: Use strings for large numbers
{
  "id": "9007199254740993",
  "bigint": "18446744073709551615"
}
```

#### 4. **No Binary Data Support**

```json
{
  "avatar": "data:image/png;base64,iVBORw0KGgoAAAANSUhEUg..."
}
```

Binary data must be Base64-encoded, which increases size by ~33%.

**Alternatives:**
- Send binary data separately (multipart/form-data)
- Use URL references: `"avatar": "https://cdn.example.com/avatar.png"`

#### 5. **No Reference/Pointer Support**

```json
{
  "user1": {
    "name": "John",
    "manager": {
      "name": "Jane",
      "title": "Manager"
    }
  },
  "user2": {
    "name": "Bob",
    "manager": {
      "name": "Jane",
      "title": "Manager"
    }
  }
}
```

Jane's data is duplicated. No way to reference the same object.

**Workaround:** Use IDs and relationships:
```json
{
  "users": [
    { "id": 1, "name": "John", "manager_id": 3 },
    { "id": 2, "name": "Bob", "manager_id": 3 }
  ],
  "managers": [
    { "id": 3, "name": "Jane", "title": "Manager" }
  ]
}
```

#### 6. **Verbose for Repeating Structures**

```json
{
  "users": [
    {
      "firstName": "John",
      "lastName": "Doe",
      "email": "john@example.com"
    },
    {
      "firstName": "Jane",
      "lastName": "Smith",
      "email": "jane@example.com"
    }
  ]
}
```

Keys are repeated for every object. CSV would be more compact:

```csv
firstName,lastName,email
John,Doe,john@example.com
Jane,Smith,jane@example.com
```

#### 7. **No Streaming Support**

JSON must be fully parsed before use:

```javascript
// Must load entire JSON into memory
const data = JSON.parse(hugeJsonString); // Could be 100MB!
```

For very large datasets, this can cause memory issues.

**Alternatives:**
- JSON Lines (JSONL) - one JSON object per line
- Streaming JSON parsers (JSONStream, Oboe.js)

#### 8. **No Metadata/Type Information**

```json
{
  "timestamp": 1678176000000
}
```

Is this milliseconds? Seconds? A UTC timestamp? No way to know without documentation.

**Workaround:** Use descriptive field names:
```json
{
  "timestamp_ms": 1678176000000,
  "created_at_iso": "2026-03-07T10:00:00Z"
}
```

#### 9. **Security Concerns**

**eval() vulnerability (legacy):**
```javascript
// ❌ NEVER DO THIS - Security vulnerability!
const data = eval('(' + jsonString + ')');

// Attacker could inject:
// '({ name: "John" }); maliciousCode(); ({})'

// ✅ ALWAYS use JSON.parse()
const data = JSON.parse(jsonString);
```

**JSON injection in constructing JSON:**
```javascript
// ❌ Vulnerable to injection
const json = `{"search":"${userInput}"}`;

// If userInput = '", "admin": true, "x":"'
// Result: {"search":"", "admin": true, "x":""}

// ✅ Use JSON.stringify()
const json = JSON.stringify({ search: userInput });
```

#### 10. **Strict Parsing**

```json
{
  "name": "John",
}
```

A single trailing comma breaks the entire JSON. No tolerance for errors.

## JSON Best Practices

### 1. **Use Consistent Naming Convention**

```json
// ✅ Good: snake_case (recommended for APIs)
{
  "user_id": 42,
  "first_name": "John",
  "created_at": "2026-03-07T10:00:00Z"
}

// ✅ Also good: camelCase (common in JavaScript)
{
  "userId": 42,
  "firstName": "John",
  "createdAt": "2026-03-07T10:00:00Z"
}

// ❌ Bad: Mixed conventions
{
  "user_id": 42,
  "firstName": "John",
  "created-at": "2026-03-07T10:00:00Z"
}
```

### 2. **Use ISO 8601 for Dates**

```json
{
  "created_at": "2026-03-07T10:00:00Z",
  "updated_at": "2026-03-07T15:30:00+01:00",
  "date_only": "2026-03-07"
}
```

### 3. **Use Null for Missing Values**

```json
{
  "name": "John Doe",
  "middle_name": null,
  "phone": null
}
```

**Don't omit fields** - makes parsing easier.

### 4. **Avoid Deep Nesting**

```json
// ❌ Too deep (hard to work with)
{
  "user": {
    "profile": {
      "personal": {
        "contact": {
          "email": "john@example.com"
        }
      }
    }
  }
}

// ✅ Flatten structure
{
  "user": {
    "email": "john@example.com",
    "profile": {
      "bio": "Developer"
    }
  }
}
```

**Limit nesting to 2-3 levels max.**

### 5. **Pretty Print in Development**

```javascript
// Development: Pretty print (readable)
JSON.stringify(data, null, 2);
// {
//   "name": "John",
//   "age": 30
// }

// Production: Minified (smaller)
JSON.stringify(data);
// {"name":"John","age":30}
```

### 6. **Validate with JSON Schema**

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "User",
  "type": "object",
  "properties": {
    "id": {
      "type": "integer",
      "minimum": 1
    },
    "name": {
      "type": "string",
      "minLength": 1,
      "maxLength": 100
    },
    "email": {
      "type": "string",
      "format": "email"
    },
    "age": {
      "type": "integer",
      "minimum": 0,
      "maximum": 150
    }
  },
  "required": ["id", "name", "email"]
}
```

### 7. **Use Arrays for Collections**

```json
// ✅ Good: Array of objects
{
  "users": [
    { "id": 1, "name": "John" },
    { "id": 2, "name": "Jane" }
  ]
}

// ❌ Bad: Object with numbered keys
{
  "users": {
    "0": { "id": 1, "name": "John" },
    "1": { "id": 2, "name": "Jane" }
  }
}
```

### 8. **Avoid Redundant Wrapping**

```json
// ❌ Unnecessary wrapping
{
  "data": {
    "user": {
      "name": "John"
    }
  }
}

// ✅ Direct structure
{
  "user": {
    "name": "John"
  }
}
```

## JSON Schema Validation

### Basic Schema

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "name": {
      "type": "string",
      "minLength": 1,
      "maxLength": 100
    },
    "age": {
      "type": "integer",
      "minimum": 0
    },
    "email": {
      "type": "string",
      "format": "email"
    },
    "status": {
      "type": "string",
      "enum": ["active", "inactive", "pending"]
    }
  },
  "required": ["name", "email"]
}
```

### Validation in JavaScript

```javascript
const Ajv = require('ajv');
const ajv = new Ajv();

const schema = {
  type: 'object',
  properties: {
    name: { type: 'string' },
    age: { type: 'number', minimum: 0 }
  },
  required: ['name']
};

const validate = ajv.compile(schema);

// Valid data
const valid = validate({ name: 'John', age: 30 });
console.log(valid); // true

// Invalid data
const invalid = validate({ age: 30 }); // Missing 'name'
console.log(invalid); // false
console.log(validate.errors);
// [{ 
//   keyword: 'required',
//   message: "must have required property 'name'"
// }]
```

        "type": "string"

      "type": "object",

        "street": { "type": "string" },

## JSON vs Other Formats

### JSON vs XML

| Feature | JSON | XML |
|---------|------|-----|
| **Size** | Smaller (40-50% less) | Larger (verbose) |
| **Readability** | Very readable | Less readable |
| **Parsing Speed** | Fast (10x faster) | Slower |
| **Data Types** | 6 types | Only strings |
| **Comments** | ❌ No | ✅ Yes |
| **Attributes** | ❌ No | ✅ Yes |
| **Namespace** | ❌ No | ✅ Yes |
| **Schema** | JSON Schema | XSD |
| **Best For** | APIs, web apps | Documents, legacy systems |

**Example:**

```json
{
  "book": {
    "title": "Learning JSON",
    "author": "John Doe",
    "year": 2026
  }
}
```

```xml
<book>
  <title>Learning JSON</title>
  <author>John Doe</author>
  <year>2026</year>
</book>
```

### JSON vs YAML

| Feature | JSON | YAML |
|---------|------|------|
| **Readability** | Good | Excellent |
| **Size** | Compact | Similar |
| **Comments** | ❌ No | ✅ Yes |
| **References** | ❌ No | ✅ Yes |
| **Data Types** | Basic | Rich (dates, custom) |
| **Parsing Speed** | Very fast | Slower |
| **Best For** | APIs, data exchange | Configuration files |

**Example:**

```json
{
  "database": {
    "host": "localhost",
    "port": 5432,
    "credentials": {
      "username": "admin",
      "password": "secret"
    }
  }
}
```

```yaml
# YAML equivalent
database:
  host: localhost
  port: 5432
  credentials:
    username: admin
    password: secret
```

### JSON vs Protocol Buffers

| Feature | JSON | Protocol Buffers |
|---------|------|------------------|
| **Size** | Larger | Much smaller (5-10x) |
| **Speed** | Fast | Very fast (5-10x) |
| **Readability** | Human-readable | Binary (not readable) |
| **Schema** | Optional | Required |
| **Best For** | Public APIs | Internal services, performance-critical |

## Common Pitfalls and Solutions

### 1. **Trailing Commas**

```json
// ❌ Invalid
{
  "name": "John",
  "age": 30,
}

// ✅ Valid
{
  "name": "John",
  "age": 30
}
```

### 2. **Circular References**

```javascript
// ❌ Problem: Circular reference
const obj = { name: 'John' };
obj.self = obj;

JSON.stringify(obj); // Error: Converting circular structure to JSON

// ✅ Solution: Remove circular references
const { self, ...cleanObj } = obj;
JSON.stringify(cleanObj);
```

### 3. **Undefined Values**

```javascript
// ❌ Problem: undefined becomes null or is omitted
JSON.stringify({ name: 'John', age: undefined });
// Result: '{"name":"John"}'

// ✅ Solution: Use null explicitly
JSON.stringify({ name: 'John', age: null });
// Result: '{"name":"John","age":null}'
```

### 4. **Date Serialization**

```javascript
// ❌ Problem: Dates become strings
const data = { created: new Date() };
JSON.stringify(data);
// '{"created":"2026-03-07T10:00:00.000Z"}'

// ✅ Solution: Parse back to Date when needed
const parsed = JSON.parse(jsonString);
parsed.created = new Date(parsed.created);
```

### 5. **Precision Loss with Large Numbers**

```javascript
// ❌ Problem: Precision lost
const bigNum = 9007199254740993;
JSON.parse(JSON.stringify(bigNum)); // 9007199254740992 (wrong!)

// ✅ Solution: Use strings for large numbers
const data = { id: '9007199254740993' };
```

## Summary

### JSON Strengths

✅ **Lightweight** - 40% smaller than XML
✅ **Human-readable** - Easy to understand
✅ **Fast parsing** - 10x faster than XML
✅ **Universal support** - Every language has JSON parsers
✅ **Native JavaScript** - No external library needed
✅ **Simple syntax** - Only 6 data types
✅ **Perfect for APIs** - Industry standard for REST APIs
✅ **Schema validation** - JSON Schema for validation
✅ **Wide adoption** - Used by all major APIs

### JSON Weaknesses

❌ **No comments** - Cannot add explanations
❌ **No date type** - Dates as strings (ISO 8601)
❌ **Limited precision** - Large numbers lose precision
❌ **No binary data** - Must use Base64 encoding
❌ **No references** - Data duplication
❌ **Verbose for tables** - Repeating keys
❌ **No streaming** - Must parse entire document
❌ **No metadata** - Type information must be external
❌ **Strict parsing** - One error breaks everything

### When to Use JSON

✅ **REST APIs** - Perfect choice
✅ **Configuration files** - Simple configs (package.json)
✅ **Data storage** - NoSQL databases (MongoDB)
✅ **Browser storage** - localStorage, sessionStorage
✅ **Data exchange** - Between services, microservices
✅ **Logging** - Structured logs
✅ **Webhooks** - Event notifications

### When NOT to Use JSON

❌ **Large binary data** - Use multipart/form-data
❌ **Extreme performance needs** - Use Protocol Buffers
❌ **Streaming large datasets** - Use JSONL or streaming parsers
❌ **Complex documents** - XML might be better (namespaces, attributes)
❌ **Configuration with comments** - YAML or TOML might be better
